Trust Map

信任規則總表

這一章專門回答一個常被講得太含糊的問題：GCD 的 trust 到底在擋什麼？它擋的是整個 runtime，還是特定類型的工作區來源？

先講最重要的現況

GCD 的 trust 目前不是一個包山包海的總開關，而是比較接近「哪些 repo-owned / workspace-scoped 設定 可以進入 runtime」的前置 gate。這個功能已經很有用，但還不是完全完全統一的機制，因為像 .gcd/mcp.json、global plugins、session reuse、MEMORY.md 就不完全走同一套規則。

為什麼 trust 值得單獨成章

很多文件談 trust 時，只會說「有 trusted folders」或「untrusted repo 會被限制」。但對工程師來說，真正重要的是更細的問題：到底哪些檔案會被相信？哪些設定可以改變 prompt？哪些能力就算在 untrusted workspace 裡還是會繼續存在？

如果這些問題講不清楚，讀者很容易把 trust 誤解成 sandbox，或者以為所有 workspace 來源都會被全部擋下。GCD 的原始碼剛好很適合拿來教這件事，因為它既有明確的 信任檢查，也有幾個很值得如實標記的不對稱路徑。

如果你問的是 真正該回去看哪一層 原因
這個 repo 的規則能不能進 prompt Trust + instructions / commands / skills loader 這是 trust 最直接影響的地方。
工具能不能碰磁碟或進容器 Sandbox 這不是 trust 在回答的問題。
某個 tool call 要不要人工批准 Permission 這是 approval 流程，不是 workspace 信任問題。
某些 workspace 設定為什麼沒被 trust 擋下 各 loader 的實際連接位置 GCD 目前不是所有來源都共用同一個 gate。

TrustState 目前有哪幾種狀態

TrustStore::evaluate() 目前會回傳四種常見可見狀態，再加上一種重要但常被忽略的 disabled。這些狀態不只是 UI 字串，而是會直接影響 loader 行為。

如何出現 對 workspace config 的意義
trusted 有明確的 trusted 規則命中目前 workspace。 workspace config 可進主要流程。
trusted-via-parent 命中的是父層目錄的 parent-trusted 規則。 workspace config 同樣可進主要流程。
untrusted 命中 untrusted 規則。 會限制大部分 workspace-scoped config。
unknown trust 功能開啟，但找不到任何匹配規則。 目前也會被視為受限制狀態。
disabled GCD_TRUST 被關閉。 整條 信任檢查 失效，所有 loader 都當成未受限制。
一個很重要的細節

restricts_project_config() 的條件不是「只擋 untrusted」，而是「只放行 trusted 與 trusted-via-parent」。換句話說，只要 trust 功能有開，而且不是這兩種狀態，workspace config 就會被縮減。這也包含了 unknown。

Parent 規則與最長匹配，決定了 trust 不是平面列表

TrustStore 不是遇到第一條規則就停下來，而是會找所有可匹配路徑中「最深的一條」。這意味著 trust 更像路徑樹的覆寫規則，而不是簡單白名單。

範例：最長匹配如何影響結果 
trusted-folders.txt
parent   /Users/alice/work
untrusted /Users/alice/work/3rd-party/danger-repo

情境 A
workspace = /Users/alice/work/my-app
=> trusted-via-parent

情境 B
workspace = /Users/alice/work/3rd-party/danger-repo
=> untrusted

重點：
不是先看到 parent-trusted 就結束，
而是更深的 untrusted 規則會覆蓋前者。

這種設計很實用，因為它允許你把大範圍開成可用，再對特定高風險子目錄做更嚴格的覆蓋。所以，文件在描述 trust 時不能只說「有沒有 trusted folder」，而要說清楚它其實是一套路徑優先順序。

真正重要的地方：哪些來源會被 信任檢查 擋下

GCD 最值得教學的，不是它有 trust 這個詞，而是不同來源進 執行時 的方式真的不一樣。下面這張表，就是整站最核心的 trust matrix 之一。

來源 是否受 信任檢查 影響 GCD 目前行為
Built-in tools 否 每次 執行時 都存在，不看 workspace trust。
Built-in / env / global providers 否 built-in profiles、環境變數 providers、全域 providers.conf 都會載入。
workspace providers.conf 是 只有在 trusted / trusted-via-parent 時才載入。
workspace active-provider.txt 是 untrusted / unknown workspace 不會讀，也不能用 workspace scope 寫入。
AGENTS.md / GEMINI.md / CLAUDE.md / GCD.md 是 受限時直接回空 bundle，不進 prompt。
commands catalog 是，但方式比想像更強 一旦受限，整份 catalog 直接回空，不只 workspace commands 消失，連 global commands 也不載。 / 行為值得特別標註
skills catalog 是，但方式比想像更強 和 commands 一樣，受限時整份 catalog 直接回空。 / 行為值得特別標註
global plugins 否 無論 workspace trust 如何都會載入。
workspace plugins 是 只在未受限制時載入。
workspace hooks .gcd/hooks.json 目前看起來不是同一套 gate run_agent() 內的 HooksConfig::load() 會直接讀 workspace hooks。
.gcd/mcp.json 目前看起來不是同一套 gate run_agent() 直接對 workspace root 呼叫 load_mcp_tools()。
session resume / fork 否 只檢查 session 屬於同一個 workspace，不看 trust 狀態。
MEMORY.md 目前尚未形成主要流程含義 有讀寫介面，但沒有穩定進 prompt / 代理迴圈，trust 規則也尚未被主要流程定義。

最容易被忽略的細節：commands / skills 和 plugins 的 trust 行為不一樣

這裡有一個非常值得教學化指出的差異。很多人會直覺以為「untrusted workspace 只是不讀 workspace-local 設定，global 設定仍可用」，但 GCD 對不同來源的做法並不完全相同。

來源類型 untrusted / unknown workspace 下會發生什麼 對使用者體感的影響
Commands 整份 command catalog 變空。 你連 global command 都看不到。
Skills 整份 skill catalog 變空。 你連 global skill 都不能選。
Plugins global plugins 照常存在，只是不讀 workspace plugins。 某些擴充工具仍可能出現。
Providers built-in / env / global provider 仍存在，只是不讀 workspace provider 設定。 agent 仍可跑，但專案特化 provider 不會生效。
Hooks global hooks 與 workspace hooks 都可能仍被載入。 工作區層政策未必像 commands / skills 那樣被一起限制。

這種不對稱本身不一定是錯，但它很重要所以需要寫清楚。因為如果文件只說「untrusted 會限制 workspace config」，讀者可能不會想到 commands / skills 的限制比 plugins 更強。

三個具體例子，幫你建立直覺

01

同一個陌生 repo，在 trust 關閉時會突然變得很“完整”

如果 GCD_TRUST=0，status_label() 會變成 disabled。這時 instructions、workspace providers、workspace commands、workspace skills、workspace plugins 都不再被 信任檢查 擋下。這對測試很方便，但對陌生 repo 風險也最高。

02

不可信 repo 下，global plugin 還在，但 global command 反而不見

這是目前 GCD 很值得被文件化的一個行為差異。對讀者來說，這能幫助他們理解 信任檢查 不只是「global vs workspace」，還牽涉各 catalog loader 是在哪裡做 early return。

03

工作接續 不看 trust，而是看 workspace 身分

load_session_for_workspace() 只確認 session 屬於同一個 workspace root，不會再額外檢查 trusted / untrusted。這表示 session 是以工作區一致性為主，而不是把 trust 當作 resume 的必要條件。

範例：同一個工作區，在不同 trust 狀態下的差異 
情境：/review src/app.rs

Case A: trusted-via-parent
- AGENTS.md 會進 prompt
- workspace providers 會載入
- commands / skills 會可用
- workspace plugins 會可用

Case B: unknown
- instructions bundle 變空
- workspace providers 不載
- commands / skills catalog 直接變空
- global plugins 仍存在
- MCP 目前仍要保守看待是否會被一起擋下

Case C: disabled
- 信任檢查 失效
- 所有 workspace 設定都會照常進主要流程

MCP 與 Hooks，為什麼是 trust 含義中最值得特別標記的例外

從 教學範例 的角度來看，MCP 與 hooks 很值得被拿出來單講，因為它們正好示範了一種常見的開發中的狀態：系統已經有 信任檢查，但新擴充路徑還沒有完全併進同一套含義。

載入位置不同

commands / skills / instructions / plugins 都是在 App::exec 前半段配合 TrustState 載入；MCP 與 hooks 則是在更靠近 執行時 的位置，直接用 workspace root 去找 .gcd/mcp.json 與 .gcd/hooks.json。

設定來源同樣屬於 workspace

從使用者觀點看，.gcd/mcp.json 與 .gcd/hooks.json 顯然也都是 repo-owned config；也正因如此，它們沒有和其他 workspace config 共用同一套 gate，才更值得文件保守描述。

影響層級很高

MCP 不是單純讀一份說明文件，而是可能啟動外部子程序、動態引入新的工具集合；hooks 則能在工具前後插入額外政策與阻擋。這代表它們如果未與 trust 含義對齊，文件就不能用過度簡化的口氣帶過。

這裡最誠實的說法是什麼

最誠實的描述不是「MCP 與 hooks 完全不受 trust 影響」，而是「目前 .gcd/mcp.json 與 .gcd/hooks.json 沒有明顯走進和 instructions / commands / skills / workspace providers / workspace plugins 完全一致的 信任檢查，因此應保守標成 Implemented 的邊界」。

MEMORY.md 則是另一種不同的問題

Memory 與 MCP 的 trust 整合狀態不同。MCP 是主要流程有實際路徑且 trust 含義已對齊；MEMORY.md 則已透過 read_memory 接入 App::exec 與 提示詞組裝 主要流程，其內容會在每次執行時注入 prompt。

來源 目前主要問題 對 trust 文件的意思
MCP 主要流程有載入，但 trust 對齊未完全一致。 要標成例外與中間態。
MEMORY.md 主要流程尚未穩定讀取。 不要過度延伸 trust 含義，先標 Implemented 即可。

對貢獻者最有用的設計問題

如果你之後要在 GCD 加新的 workspace source，這一章其實提供了一套很好用的檢查表。不要先問「這功能酷不酷」，先問它應不應該進同一套 trust 含義。

這個來源是 user-owned global config，還是 repo-owned workspace config？

如果是後者，就很難完全跳過 trust 問題。

它只影響 observability，還是會改變 prompt / tools / provider / 執行時 行為？

越接近 執行時 主要流程，越需要明確 trust 含義。

它是否和既有 workspace config 共享同一個風險等級？

如果和 commands、skills、workspace providers 一樣都能改變 agent 行為，卻不走同一套 gate，就該至少在文件裡誠實標示原因。

它在 untrusted / unknown / disabled 三種狀態下，各自會怎麼表現？

如果這三種情境答不出來，通常表示設計還沒有真正定型。

把這一章濃縮成一句話

Trust 在 GCD 裡主要是在管「哪些工作區來源可以改變 agent」

它不是 sandbox，也不是 approval；它更像一張 source boundary 地圖。Instructions、workspace providers、workspace plugins 類型已經清楚接上，commands / skills 有更強的整體被限縮行為，MCP 仍是值得保守標記的例外，而 MEMORY.md / USER.md 已透過 memory 工具接入主要流程並含安全掃描。把這些差異講清楚，讀者才會真的懂 trust，而不是只記得一個標籤名。
