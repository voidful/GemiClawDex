驗證迴圈 — 讓 Agent 自己檢查自己的工作

生產級 harness 的核心 pattern：反覆執行直到通過驗證。對照 GCD 的 coordinator 和 Hermes Agent 的 delegate_tool。

一句話解釋

AI 模型會犯錯。Harness 的責任不是讓模型不犯錯，而是偵測到錯誤後自動重試。這個「執行 → 檢查 → 重試」的迴圈，就是驗證迴圈。

為什麼一次性生成不夠

即使是最強的模型，單次生成的成功率也不是 100%。但如果你讓模型嘗試三次，每次都檢查結果，總成功率會大幅提高。這不是理論，是 OpenAI 和 Anthropic 在 2026 年公開分享的工程經驗。

OpenAI 把這個 pattern 叫做 Ralph Wiggum Loop：一個 bash 腳本反覆啟動 AI agent，每次都給它全新的 context，直到所有任務通過驗證。名字來自辛普森家庭的角色，意思是「即使是不太聰明的 agent，只要有好的迴圈，也能完成工作」。

GCD 的 Coordinator 怎麼做

GCD 的 spawn_agent 工具支援帶依賴的任務批次，由 tools/coordinator.rs（932 行）實作。這就是 GCD 版本的驗證迴圈。

spawn_agent 任務批次範例 
{
  "tasks": [
    { "name": "scan", "task": "掃描模組缺口並列出要改的檔案" },
    { "name": "design", "task": "提出最小可行設計", "depends_on": ["scan"] },
    { "name": "implement", "task": "依設計實作變更", "depends_on": ["design"] },
    { "name": "review", "task": "檢查回歸風險與測試缺口", "depends_on": ["implement"] }
  ],
  "strategy": "parallel",
  "max_concurrency": 2
}

Coordinator 的關鍵能力：

1

依賴驗證

每個 task 可以聲明 depends_on。Coordinator 會驗證依賴圖，偵測循環依賴。

2

Ready Batch 平行執行

沒有未完成依賴的任務會被分成一批同時執行，受 max_concurrency 限制。

3

失敗傳播

如果一個 task 失敗，依賴它的所有下游 task 會被標記為 blocked，不會浪費 token 嘗試。

4

事件證據

每個 coordinator 事件（started, batch started, task completed, blocked）都會記錄在 session JSONL 裡，可以用 sessions replay 回放。

原始碼：crates/gcd-core/src/tools/coordinator.rs

Anthropic 的三層架構

Anthropic 在 2026 年公開了一個更精緻的架構，靈感來自 GAN（生成對抗網路）：

P
Planner — 把大任務拆成小步驟，產出 feature list。
G
Generator — 執行每個步驟，產出程式碼。
E
Evaluator — 檢查 Generator 的輸出，決定是否通過。不通過就回到 Generator。

GCD 的 coordinator 可以模擬這個架構：scan = Planner，implement = Generator，review = Evaluator。差異在於 GCD 的 review task 目前是單次執行，不會自動觸發 re-implementation。

Hermes Agent 的做法

Hermes Agent 的 delegate_tool.py（46K 行）和 subagent-driven-development skill 實作了更完整的版本：

  • 每個 subagent 都有全新的 context（避免前一個 task 的 context 污染）
  • 兩階段 review：先檢查 spec compliance，再檢查 code quality
  • 自動修復迴圈：review 不通過 → 修正 → 再 review

自己設計驗證迴圈的思考框架

問題你的回答決定了迴圈的設計
什麼算「通過」？ 測試通過？Lint 清零？LLM-as-judge 給分 ≥ 8？人類確認？
什麼時候該放棄重試？ 最多重試 N 次？超過 token 預算？連續失敗 pattern 相同？
每次重試要給什麼 context？ 上次的錯誤訊息？全新 context？壓縮後的歷史？
用 Computational 還是 Inferential sensor？ 測試套件（computational）便宜但只能檢查已知行為。LLM-as-judge（inferential）能發現未預期的問題但成本高。

Fowler 的 Guides vs Sensors 四象限

把 GCD 的組件放進 Martin Fowler 的分類，會長這樣：

Guide（引導行為）Sensor（檢查輸出）
Computational cargo clippy（linter）、Cargo.toml 的依賴限制 cargo test（test suite）、CI pipeline
Inferential AGENTS.md、MEMORY.md、Skill descriptions coordinator 的 review task、session_search 回溯
