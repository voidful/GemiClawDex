工具擴充方式

GCD 有三種工具：內建的、Plugin 擴充的、MCP 連接的。它們載入方式不同、安全規則也不同。

先講最重要的 實況檢查

GCD 的「可擴充性」不是一條單線能力。built-in tools、plugin tools、MCP tools 雖然最後都會出現在同一個 tool registry 裡，但它們的載入來源、信任規則、執行方式、命名規則與安全意義並不一樣。如果把它們都簡化成「外部工具」，文件就會不準確。

先把三層工具面分開看

層次 它從哪裡來 它怎麼進 runtime
Built-in tools tools/registry.rs 硬編碼 每次 run_agent() 都一定存在
Plugin tools 全域 ~/.gcd/plugins 與 workspace .gcd/plugins App::exec 先載入，再傳進 run_agent()
MCP tools workspace .gcd/mcp.json run_agent() 啟動時動態連線與發現

工具註冊表是怎麼合併的

GCD 不是讓三種工具各走各的執行路徑，而是最後都合併進同一份工具清單。builtin_tools() 先建立 11 個內建工具，接著 all_tools(plugin_tools, mcp_tools) 會依序把 plugins 與 MCP tools append 到後面。這表示「所有工具都長得像 Tool trait」，但不代表它們擁有相同的風險程度。

教學性示意：工具合併順序 
1. built-in tools
   - read_file
   - write_file
   - list_dir
   - shell
   - search_files
   - fetch_url
   - apply_patch
   - spawn_agent
   - memory          ← Hermes-inspired：雙重記憶
   - skill_manager   ← Hermes-inspired：技能學習迴圈
   - session_search  ← Hermes-inspired：對話回溯

2. plugin tools
   - 由 PluginCatalog 載入

3. MCP tools
   - 由 .gcd/mcp.json 啟動伺服器後動態發現

這裡有一個很重要所以需要寫清楚的細節：execute_tool_call() 會用 tools.iter().find(...) 找第一個同名工具。換句話說，如果發生命名衝突，目前是靠註冊順序決定誰先被找到，而不是有一套明確的名稱衝突檢查。

這個順序的實際意義

Built-in tools 會先於 plugins 被找到，plugins 又會先於 MCP。換句話說，如果有人真的用相同名字定義 plugin 或 MCP tool，它不一定能覆蓋前面的工具，甚至可能只是被悄悄 shadow。註解說名稱應唯一，但目前沒有強制驗證。

Built-in tools：最完整，也最像 GCD 的正式含義

內建工具是目前最成熟、也最接近 GCD 官方含義的工具層。它們共享 ToolContext、多數會檢查 sandbox，並與 permission / hooks / session events 有最完整的連接。網站前面那些對 read_file、write_file、apply_patch、shell、spawn_agent 的分析，基本上都是以 built-in path 為準。新增的 memory、skill_manager、session_search 三個工具則操作 ~/.gcd/ 下的持久知識檔案，不直接影響工作區原始碼。

B1

命名穩定

工具名稱固定，provider adapters 與 permission 邏輯都以這些名稱為基礎。

B2

Sandbox 含義最清楚

至少 read-only、workspace-write、container 的主要差異，都是從 built-in tools 看得最清楚。

B3

Permission 對照最完整

suggest、auto-edit、full-auto 的設計主要就是圍繞這批內建工具定義出來的。

Plugins：比較像本地命令擴充，不是新的協定層

Plugins 的定位，和 MCP 很不一樣。它們不是去連外部協定，而是把一個 JSON 定義包成新的 Tool。定義中包含工具名稱、描述、JSON Schema、命令模板、可選工作目錄與輸出上限。執行時則會把命令切成 program + args，再做 placeholder 置換，最後用 Command::new(program).args(...) 執行。

範例：一個 plugin JSON 
{
  "name": "lint_code",
  "description": "Run ESLint on a file",
  "parameters": {
    "type": "object",
    "properties": {
      "file": { "type": "string" }
    },
    "required": ["file"]
  },
  "command": "npx eslint {{file}}",
  "cwd": "frontend",
  "max_output_bytes": 8192
}

這個設計有一個很實用的優點：它不是把整串命令交給 sh -c，而是先做簡單 tokenization，再把參數以字面值放進 argv。這大幅降低了傳統 shell template 最直接的字串插值風險，也表示 plugin 比「把模板塞進 shell 工具」更收斂一些。

Plugin 層的優點 對教學與工程有什麼幫助
不走 sh -c 降低最直接的 shell string injection 風險。
可以定義自己的 schema 讓 provider 看見一個具名、具參數結構的新工具，而不是模糊文字說明。
可指定 cwd 適合把某些前端或子模組工作封裝成本地工具。
全域與 workspace 可分開載入 兼顧個人習慣與專案特化。

但 plugin 的安全意義，目前不能講得太滿

這裡有兩個很重要的 實況檢查。第一，plugin 執行時目前沒有像 built-in shell 那樣依 ctx.sandbox 去改走 container，也沒有明確阻擋 read-only。第二，permission prompt 的 needs_confirmation() 目前只認得 write_file、apply_patch、shell 這幾個名字，因此一般 plugin tool 預設不會自動落入那套批准含義。

面向 Built-in shell 一般 plugin tool
在 read-only 下 會直接拒絕 目前程式碼裡沒有同等檢查
在 container 下 可改走 Docker / Podman 目前仍直接在 host 上執行
Permission prompt 依名稱命中 shell 邏輯 除非名稱剛好撞到既有高風險工具，否則不會自動套入
這不代表 plugins 不能用，而是含義還沒完全統一

Plugins 已經能在主要流程工作，但如果要把它們描述成「與 built-in tools 共享同一套 sandbox / permission 安全設計」，目前會過度樂觀。比較誠實的說法是：plugins 目前是功能可用、邊界仍需精煉的實驗中的工具層。

MCP：不是本地命令模板，而是外部工具協定橋接

MCP 路徑和 plugins 最大的差別，在於它不是直接執行一個命令模板，而是先讀 .gcd/mcp.json，啟動指定的 MCP server 子程序，做一次 initialize 握手，再跑 tools/list 發現可用工具，最後把每個外部工具包成新的 McpToolAdapter。

範例：workspace MCP 設定 
{
  "servers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": {}
    }
  }
}

發現到的 MCP 工具，最後會被命名成 mcp__{server_name}__{tool_name}。這種 namespacing 很重要，因為它避免不同 server 上同名工具直接撞名，也讓 session / replay 時能一眼看出某個工具呼叫其實來自哪個 MCP server。

範例：MCP 工具名稱會怎麼長 
server name: filesystem
tool name: read_file

=> GCD 執行時 中的工具名：
mcp__filesystem__read_file

MCP 的擴充價值，在於協定，不在於命令模板

如果說 plugins 比較像把本地命令包成工具，那 MCP 則是在說：「GCD 可以透過標準協定，向外部工具生態要能力」。這對產品邊界來說非常重要，因為它讓工具擴充不必全都塞進本地 repo，也不必全部寫成內建工具。

M1

工具發現是動態的

先啟 server，再詢問它有哪些工具，而不是本機先寫死每個工具定義。

M2

工具 schema 來自對方

GCD 會使用對方回傳的 inputSchema 生成 tool definition，而不是自己手寫。

M3

工具執行透過 JSON-RPC

呼叫的不是本地命令列，而是 tools/call 這種協定方法。

但 MCP 的 trust 與 sandbox 邊界，目前也值得保守描述

這裡有一個很關鍵、也很容易被忽略的現況：run_agent() 會直接對工作區呼叫 load_mcp_tools(&workspace_root)，而 mcp.json 的讀取目前看起來不是透過 TrustState 決定。換句話說，MCP workspace config 和 commands / skills / workspace providers / workspace plugins 的 trust gating 含義，並不完全一致。

工作區擴充來源 目前是否明確受 信任檢查 影響 說明
.gcd/commands 是 untrusted 時 catalog 會變成空的。
.gcd/skills 是 和 commands 類似。
.gcd/providers.conf 是 untrusted 時不讀 workspace provider config。
.gcd/plugins 是 workspace plugins 只在 trusted workspace 載入。
.gcd/mcp.json 目前看起來未完整對齊 ；目前是由 run_agent() 直接讀 workspace config。
這是文件裡一定要誠實講出的差異

如果把 MCP 直接寫成「和 commands / skills 一樣受 trust 管理」，那就會高估目前實作的一致性。比較準確的說法是：MCP 已有可用 client 與動態工具發現，但 workspace config 的 trust 含義還沒有和其他擴充來源完全對齊。

Built-in、Plugin、MCP 三者最關鍵的差異

面向 Built-in Plugin MCP
能力來源 GCD 原始碼內建 本地 JSON 定義 + 本地命令 外部 server 透過協定宣告
命名規則 固定名稱 由 JSON 的 name 決定 mcp__server__tool namespaced
Sandbox 含義 最完整 目前未完全對齊 built-in 主要依 MCP server 自身與 GCD 包裝邊界
Permission 含義 明確對齊 shell / write_file / apply_patch 目前不會自動套進同一套高風險分類 目前也沒有對應 built-in 的細緻分類
Trust 邊界 不涉及載入 workspace plugins 受 trust 影響 workspace config trust 含義尚未完全對齊

如果你是貢獻者，怎麼判斷該做 plugin 還是 MCP

01

如果只是本地命令封裝，先想 plugin

像 lint、formatter、repo 專用腳本、子模組工具鏈，如果主要是包裝本地命令與 schema，plugin 常常就夠了。

02

如果你要的是外部工具生態，才去想 MCP

MCP 更適合要與外部 server 協定互通、工具集合是動態發現、且不想每個工具都手刻成本地 JSON 的情境。

03

如果能力已是核心完整功能，優先考慮 built-in

像 read_file、apply_patch、shell 這種會深度影響 permission、sandbox、events 的能力，長期來看還是更適合留在 built-in 層。

實況檢查：最值得記住的四件事

三層工具面都能工作，但完成度不同

Built-in 最完整；plugins 可用但安全意義尚未完全統一；MCP 已能連線與發現工具，但 client / 信任規則仍要保守看待。

名稱衝突目前主要靠約定，不靠強制驗證

同名工具如果真的出現，目前偏向由註冊順序決定誰先被找到。

Plugin 與 MCP 不是「另一種 shell」

Plugins 是本地命令封裝；MCP 是協定橋接；兩者都不該和 built-in shell 混成一種含義。

Trust、sandbox、permission 在擴充層還沒有完全同構

這就是為什麼文件要用 Implemented 而不是一筆帶過「支援擴充」。

把這一章濃縮成一句話

可擴充性不是功能空談，而是邊界管理

GCD 確實有 built-in、plugin、MCP 三層工具面，這很有價值；但它們不是同一種東西，也還沒有完全共享同一套 trust、sandbox、permission 含義。把這條邊界講清楚，比把「支援 MCP / plugins」寫成一行賣點更重要。
