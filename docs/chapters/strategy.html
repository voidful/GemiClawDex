參考矩陣

來源不是靈感清單，而是可追蹤的工程參考

這一章把理論、產品、社群衍生版與替代實作整理成參考來源整理，讓你看得出每個來源影響了 GCD 哪一層。

使用方式

先看主產品與方法論，再看社群反饋。對每個來源都問四個問題：它提供什麼工程參考？GCD 學了哪一部分？改造了什麼？哪些地方故意不複製？

來源研究不是收集靈感，而是建立判斷方法

如果只是把一堆 repo、文章與教學站排成連結清單，讀者通常很難真的得到什麼。這一章要做的，是把來源分成不同層級。理論與教學資料幫我們建立語言；產品與官方 repo 提供可以觀察功能範圍；開源社群專案（如 Hermes Agent）提供商業產品不會公開的實作細節；替代實作提醒我們從頭重建時容易踩的坑；社群衍生版則反映成本、相容性、本地化與安全透明度這些真實需求。

換句話說，來源在本站的角色不是「證明我們很有研究」，而是幫我們回答兩件事。第一，GCD 的設計判斷從哪裡來。第二，哪些判斷只是合理推論，哪些則能被原始碼或官方文件直接支持。

如何使用這張參考來源整理

先用理論資料建立總綱

像 外殼工程、Claude Code Book、Learn AI 助手 這類資料，幫我們先把問題框起來：要看的是 外殼，而不是單一 feature。

再用產品資料確認成熟功能範圍

Claude Code、Gemini CLI、OpenAI Codex 讓我們看到終端體驗、permission 語言、trusted folder、MCP、instruction workflow 等能力怎麼被做成可用產品。

最後用社群反饋補 實際使用情況

成本、替代端點、語言門檻、還原原始碼、安全機制的透明度，這些往往不會在官方說明裡被放太前面，但對實際採用非常關鍵。

用一個實例看「來源」如何變成「設計判斷」

這一章最重要的不是把來源背起來，而是學會一個轉化過程。下面用三個例子示範，同一則外部訊號怎麼影響到 GCD 的文件與設計裡。

外部訊號 先回答的問題 影響到 GCD 的哪裡 我們刻意保留的差異
Gemini CLI 的 trusted folders 工作區本身是否可信，應不應影響 prompt 與 workspace config？ GCD 的 信任檢查、instructions 載入條件與 workspace-scoped provider 限制。 沒有把它只做成 UI 勾選，而是讓它直接改變 執行時 行為。

從來源走到設計決策，中間其實還有兩步

01

先辨識來源提供的是哪一種訊號

它是在提供理論框架、產品能力、社群分析，還是社群需求？不同訊號的可信度與用途不同，不能混為一談。

02

再判斷這個訊號應該對應到系統的哪一層

例如 token caching 應該落在 provider / 請求格式調整 層；trusted folder 則會影響 instructions 與 workspace config 是否進 prompt。

03

最後才問它是否已被 GCD 採用

如果原始碼主要流程已經接上，就算已完成；如果只有局部連接，就標註為部分實作；如果只是方向，就說明它是研究軌。

社群衍生版提醒了我們什麼

社群最在意的問題 這反映什麼 GCD 的回應 不該複製的地方
成本與替代端點 Agent 採用受限於於 API 價格與被單一供應商綁住。 provider 清單、fallback、多供應商 profile。 為了支援一切端點而把協定層做成難維護難維護的混亂系統。
可以觀察與可執行 還原原始碼 工程師不想被看不透的 prompt 與隱藏的行為綁架。 文件站明標完成度與原始碼依據。 把 restored / reversed source 直接包裝成官方真相。
本地化與教學再編排 語言與案例會直接影響採用速度。 全站繁中重寫、閱讀路徑、比較章。 只做機械翻譯，不做結構重整。
安全界線與隱私透明度 大家會主動檢視 安全機制、telemetry、權限模型。 把 trust / sandbox / permission 當成首頁級概念。 把移除 安全機制 當成「更自由」而不是新的風險。
逆向與 從頭重建 工程師想看可以觀察主要流程，而不是只接受看不透的描述。 整站都以原始碼 實況檢查 為底，再回來談參考。 把反向推測寫成官方保證，模糊「已完成」與「推論」的界線。

參考來源整理最怕的錯誤：把不同等級的訊號混成一句話

錯誤示範：因為某個產品有 token caching，所以 GCD 也應該立刻補齊

正確做法是先問：它是 adoption 的核心瓶頸，還是第二階段的成本優化？再回頭看 GCD 主要流程是否已經準備好承接這層複雜度。

錯誤示範：社群 還原原始碼 說某能力存在，所以可以當官方真相引用

正確做法是把它標成 Implemented 或 Observed，並且只拿來解釋社群關注點，不拿來當 GCD 事實依據。

錯誤示範：某篇教學文章把一個概念講得漂亮，就直接當作本專案已實際做出來設計

正確做法是回到原始碼，確認這個概念究竟對應到哪條主要流程、哪個資料結構、哪個 執行時 閘門。

來源越多，越需要如實的標記系統

當一個教學站同時吸收理論、產品、社群分析、社群衍生版與替代實作時，最大的風險不是資訊不夠，而是把不同可信度的資訊混成同一種口氣。所以本站會明確區分「已完成」、「部分實作」與「研究方向」這三種不同的狀態。
