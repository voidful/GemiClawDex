補充章

怎麼知道你的 Agent 做得好不好

從「感覺有用」到「可量化衡量」。用 GCD 的內建工具做 agent 品質評估。

為什麼需要評估

Anthropic 在 2026 年的基準測試報告指出：光是 harness 設定的差異，就可以讓同一個模型的 benchmark 分數浮動 5 個百分點以上。換句話說，如果你不量化，你就不知道是模型的問題還是 harness 的問題。

三個評估維度

1

任務完成率

Agent 被要求做 10 件事，實際完成了幾件？每件的品質如何？這是最直觀的指標。

2

Token 效率

完成同一個任務，用了多少 token？GCD 的每次 session 都會自動追蹤 prompt / completion token 用量。更少的 token 意味著更低的成本和更快的回應。

3

工具呼叫準確率

Agent 呼叫了多少次工具？其中有多少是有效的？無效的工具呼叫（讀了不相關的檔案、執行了失敗的指令）浪費 token 和時間。

GCD 的內建可觀測性工具

工具提供什麼怎麼用
Token 追蹤 每次 session 的 prompt / completion token 用量 session 結束時自動顯示
Session JSONL 完整的事件流：每一輪對話、每一次工具呼叫、每一個結果 gcd sessions replay <id>
Coordinator 事件 批次任務的完整生命週期：started, batch, task, blocked, completed 在 session replay 中可見
Session Search 搜尋過往對話，找到類似問題的解法 session_search 工具，或在 REPL 中使用

做一次簡單的 A/B 測試

最實用的評估方法：固定任務，改變一個 harness 變數，比較結果。

A/B 測試範例 
# A 組：沒有 AGENTS.md
mv AGENTS.md AGENTS.md.bak
gcd exec "在 src/cache.rs 加入 TTL 過期測試"
# 記錄：token 用量、完成品質、工具呼叫次數

# B 組：有 AGENTS.md
mv AGENTS.md.bak AGENTS.md
gcd exec "在 src/cache.rs 加入 TTL 過期測試"
# 記錄：token 用量、完成品質、工具呼叫次數

# 比較兩組的 session replay
gcd sessions list

這就是最基本的 harness 評估。你會發現，有 AGENTS.md 的那組通常：工具呼叫更精準（因為 agent 知道專案結構）、token 用量可能更高（因為 context 更多），但任務完成品質明顯更好。

進階：設計 Eval Harness

如果你要系統化地評估 harness 效果，需要：

  • 固定測試集：5-10 個有標準答案的任務（例如「為這個函數寫測試」「修復這個 bug」）
  • 自動化執行：用 gcd exec --no-stream --json 批次執行，收集 JSONL 輸出
  • 評分標準：測試通過（binary）、程式碼品質（LLM-as-judge）、token 成本
  • 變數控制：每次只改一個 harness 組件（AGENTS.md、skill set、provider、permission level）
Hermes Agent 的做法

Hermes Agent 有一個完整的 batch_runner.py（55K 行）和 environments/ 目錄，支援 RL 訓練環境、trajectory 壓縮、基準測試。GCD 的規模不需要這麼複雜，但核心概念是一樣的：固定任務、改變配置、量化比較。
