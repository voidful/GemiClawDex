前言

什麼是 Harness Engineering？

在 AI 程式代理（Coding Agent）的世界中，模型本身只是引擎。真正決定代理品質的，是圍繞模型建構的「韁繩」（Harness）。

AI Coding Agent 的核心挑戰

當我們說「AI 寫程式」時，大多數人想像的是一個 LLM 模型接收提示、吐出程式碼。但在實際的生產級代理中，模型只佔整個系統的一小部分。真正困難且關鍵的工程在於：

Harness Engineering 的定義

Harness Engineering（韁繩工程）指的是圍繞 LLM 模型建構的一切工程基礎設施：提示詞組裝、上下文管理、工具呼叫、安全沙箱、信任邊界、會話持久化等。它是讓一個「能說話的模型」變成「能做事的代理」的全部工程。

三大設計傳承

GemiClawdex 刻意融合了三大主流 AI 編碼代理的設計精華：

來源 採納特性 對應模組
Gemini CLI REPL 互動、工具系統、GEMINI.md 上下文 tools.rs、instructions.rs
OpenAI Codex Sandbox 分級、AGENTS.md 指令、agent 工作流 config.rs、agent.rs
Claude Code Trust 邊界、Session 分支、Skill 系統 trust.rs、session.rs、skills.rs

U1

安裝與快速開始

介紹如何編譯 GemiClawdex 並進行第一次對話。

系統要求

  • Rust 1.70 或更高版本
  • 連網環境 (以取得依賴模組)
  • 準備好你的 API Key (目前支援 OpenAI, Anthropic, Gemini)

編譯與安裝

終端機
# 1. 取得原始碼並切換目錄
git clone https://github.com/your-repo/GemiClawdex.git
cd GemiClawdex

# 2. 使用 Cargo 編譯 (Release 模式以獲得最佳效能)
cargo build --release

# 3. 把執行檔移到系統路徑，方便全域使用 (選擇性)
sudo cp target/release/gcd /usr/local/bin/

第一次執行

如果你還沒設定任何環境變數，可以直接輸入 gcd。通常我們會帶上環境變數啟動，例如：

快速執行
# 設定環境變數並進入互動 REPL
GEMINI_API_KEY="AIzaSy..." gcd
如何離開 REPL？

在 gcd> 提示符號下，輸入 /exit 或短按 Ctrl+C（部分終端可能需要 Ctrl+D）即可安全退出互動模式並儲存會話紀錄。

U2

互動模式與基本操作

GemiClawdex 支援 One-off Exec 跟強大的 REPL 互動模式。

單次任務 (Exec)

如果你只想把代理當作命令列工具處理單一請求：

終端機
$ gcd exec "幫我解釋 src/agent.rs 裡面的邏輯"
$ gcd exec --provider openai-codex "在 src/ 裡面寫一個簡單的 HelloWorld"

互動模式 (REPL)

直接輸入 gcd，你就會進入與 AI 代理對話的世界：

REPL
gcd> 請分析目前的專案架構
[Agent 開始使用 list_dir 等工具探勘專案...]
Agent: 專案主要是一個 Rust workspace...

gcd> 幫我把 README.md 的錯字改一下
[Agent 呼叫 write_file 工具進行修改...]
Agent: 已經修改完成了！

常用的斜線指令 (Slash Commands)

在 REPL 中，你可以使用特定指令向系統下達控制：

  • /help：顯示幫助資訊與快速指令表
  • /overview：回報目前的工作區和系統狀態 (信任等級, 沙箱, 供應商)
  • /clear：清除目前的控制台畫面
  • /reset：清除對話上下文歷史 (新開一個乾淨的 Session)

U3

配置 LLM 供應商

GemiClawdex 支援多種強大的 LLM，你可以自由切換以符合工作需求。

系統自動偵測 (Zero Config)

最簡單的方式是將 API Key 設定為環境變數，GemiClawdex 會自動選用適合的官方引擎：

  • OPENAI_API_KEY → 對應 openai-env (預設 gpt-4o)
  • ANTHROPIC_API_KEY → 對應 anthropic-env (預設 claude-3-5-sonnet)
  • GEMINI_API_KEY → 對應 gemini-env (預設 gemini-1.5-pro)

自定義供應商設定檔

你可以透過編輯設定檔來新增模型、更改 baseUrl（例如使用第三方代理伺服器或像是 Ollama 的本機模型）。全域的設定存放在 ~/.gcd/providers.conf。

~/.gcd/providers.conf
[[provider]]
id = "my-local-ollama"
family = "OpenAiCompatible"
protocol = "OpenAIChat"
api_base = "http://localhost:11434/v1"
model = "qwen2.5-coder:32b"
supports_multimodal = false
supports_grounding = false

設定好後，可使用命令指定：gcd exec --provider my-local-ollama "寫一段 python"。

設定預設供應商

如果系統有多組 Key，或者你想固定使用自定義供應商，請設定環境變數：

~/.zshrc
export GCD_PROVIDER="my-local-ollama"

U4

客製化：指令與技能

讓代理融入你的開發流程。學習設定系統級提示詞與自定義命令。

專案層級指令：AGENTS.md / GCD.md

當你在專案根目錄建立一個 AGENTS.md 或 GCD.md 檔案時，GemiClawdex 會自動將它加到 Agent 的「系統提示詞」開頭（類似 Cursor 的 rules）。這適合：

  • 宣告專案的程式撰寫風格 (如：請使用 Rust 1.70，不要用外部套件)
  • 告知 Agent 關鍵檔案的路徑 (如：路由設定在 routes/，ORM 在 db/)

建立技能手冊 (Skills)

你可以透過撰寫 Markdown 檔案將常用的大型協作工作流打包。把技能檔放在 ~/.gcd/skills/<skill-name>/SKILL.md。

~/.gcd/skills/code-review/SKILL.md
---
description: 嚴謹的程式碼審查與重構流程
---
# 執行步驟
1. 找出最近有被修改的檔案（透過 `git status`）。
2. 分析這些檔案是否符合 SOLID 原則，並檢查潛在效能問題。
3. 如果發現問題，主動呼叫 write_file 修復。
4. 產生一個 summary.md 記錄變更與原因。

呼叫技能：gcd exec --skill code-review "請幫我審核目前變更"

建立快捷命令 (Commands)

對於重複性的特定情境需求，可以用 TOML 檔在 ~/.gcd/commands/ 來綁定 prompt 樣板：

~/.gcd/commands/lint.toml
description = "檢查程式碼規範並修復"
prompt = """
請先執行 `cargo clippy`。
如果發現警告，自動修改對應的原始碼並消除警告。
修改完成後，執行 `cargo fmt`。
"""

使用方式：在 REPL 輸入 /lint，或者在終端執行 gcd run lint。

信任機制 (Trust Mechanism) 防護

如果你的專案包含 .gcd/commands/ 目錄，只有當該專案在你的信任白名單 (~/.gcd/trusted-folders.txt) 中，代理才會載入專案專屬的命令！這是防止執行惡意代碼的重要設計。

第一章

專案概觀與設計哲學

GemiClawdex 不只是一個 CLI 工具，它是一套完整的 Harness Engineering 參考實作。

GemiClawdex 是什麼？

GemiClawdex 是一個以 Rust 編寫的終端 AI 編碼代理（Terminal AI Coding Agent），編譯為單一二進位檔 gcd。它的核心使命是：

保留 Gemini CLI、OpenAI Codex 和 Claude Code 最好的互動模式，同時不繼承 TypeScript 執行環境的全部複雜度。

六大核心設計賭注

架構文件（docs/architecture.md）明確列出了六個「Core Product Bets」——這些不是功能清單，而是設計決策：

1

Rust 為編排層

選擇 Rust 而非 TypeScript/Python，換取編譯期安全、零 GC 延遲、單一二進位部署。

2

終端優先、工作區感知

不依賴 IDE 外掛或 Web UI，直接在終端中運作，且自動發現專案根目錄。

3

可組合的三層架構

指令（Instructions）、命令（Commands）、技能（Skills）是三個獨立但可組合的層。

4

Trust & Sandbox 為一等公民

信任邊界和沙箱策略不是事後補丁，而是從第一天就嵌入架構的核心概念。

5

提示詞組裝與執行解耦

Prompt Assembly 完全獨立於 Provider Execution，讓系統在沒有網路的情況下也能測試。

6

Headless 與腳本化為一等公民

透過 --json 輸出和 Session 持久化，支援無頭執行和 CI/CD 整合。

為什麼這些設計決策很重要？

每個設計決策都對應一個真實的工程問題：

設計啟示

注意第 5 點「提示詞組裝與執行解耦」——這是 GemiClawdex 最聰明的設計之一。在大多數 AI 代理中，「組裝提示詞」和「呼叫 API」是糾纏在一起的。GemiClawdex 將它們完全分離，這意味著你可以在不連網的情況下測試整個提示詞管線。這就是真正的 Harness Engineering。

第二章

技術棧與專案結構

了解 GemiClawdex 用了哪些 crate、專案如何組織、每個檔案負責什麼。

Cargo Workspace 架構

GemiClawdex 使用 Cargo Workspace 將專案分為兩個 crate：

Cargo.toml（workspace root）
[workspace]
members = [
    "crates/gemi-clawdex-core",  # 核心邏輯庫
    "crates/gemi-clawdex-cli",   # CLI 入口二進位
]

為什麼要分兩個 crate？

架構原則

gemi-clawdex-core 擁有所有領域邏輯，不依賴 CLI 框架（clap）。gemi-clawdex-cli 只負責解析命令列參數和呈現輸出。這種分離讓 core 可以被其他前端（如 TUI、Web API）重用，也讓單元測試不需要啟動完整的 CLI。

關鍵依賴解析

Crate版本用途
tokio1.x非同步執行環境，驅動所有 API 呼叫
reqwest0.11HTTP 客戶端，與 LLM API 通訊
serde / serde_json1.0序列化框架，JSON 輸入/輸出
clap4.xCLI 參數解析（僅在 cli crate）
thiserror1.0結構化錯誤型別
anyhow1.0錯誤上下文傳遞
async-trait0.1非同步 trait 支援（用於 Tool trait）

Core 模組地圖

第三章

工作區發現機制

程式碼位置：workspace.rs（65 行）——整個系統中最小卻最基礎的模組。

問題：代理怎麼知道「專案在哪裡」？

當你在 /home/user/projects/my-app/src/utils/ 目錄下執行 gcd 時，代理需要知道專案根目錄是 /home/user/projects/my-app/，而非你所在的子目錄。這就是 Workspace Discovery 要解決的問題。

核心資料結構

workspace.rs
pub struct Workspace {
    pub root: PathBuf,        // 專案根目錄
    pub current_dir: PathBuf, // 使用者當前目錄
    pub detected_by: String,  // 是哪個標記偵測到的
}

發現演算法

Workspace::discover() 從當前目錄開始，往上逐層搜尋以下標記（按優先順序）：

標記偵測邏輯
fn detect_marker(path: &Path) -> Option<&'static str> {
    let markers = [
        (".gcd", ".gcd"), // 最高優先：GemiClawdex 自身設定
        (".git", ".git"),                     // Git 倉庫根
        ("Cargo.toml", "Cargo.toml"),         // Rust 專案
        ("package.json", "package.json"),     // Node.js 專案
    ];
    for (entry, label) in markers.iter() {
        if path.join(entry).exists() {
            return Some(*label);
        }
    }
    None
}
設計巧思

.gcd 目錄排在 .git 之前，這意味著你可以在一個 monorepo 的子專案中放置 .gcd 目錄，覆蓋 Git 根目錄的偵測。如果一路到根目錄都找不到任何標記，系統會回退（fallback）到當前目錄，用 "cwd-fallback" 標記。

衍生方法

Workspace 物件還提供了兩個便利方法，供後續模組查找工作區層級的命令和技能：

便利方法
impl Workspace {
    pub fn project_commands_dir(&self) -> PathBuf {
        self.root.join(".gcd").join("commands")
    }
    pub fn project_skills_dir(&self) -> PathBuf {
        self.root.join(".gcd").join("skills")
    }
}

第四章

組態管理與路徑系統

程式碼位置：config.rs（182 行）——定義沙箱策略、應用路徑與執行偏好。

SandboxPolicy 列舉

這是整個安全模型的基石——四個等級的沙箱策略：

config.rs — SandboxPolicy
pub enum SandboxPolicy {
    Off,             // 無沙箱——最快，最不隔離
    ReadOnly,        // 唯讀——最安全，適合程式碼審查
    WorkspaceWrite,  // 工作區寫入——允許改檔，阻擋全機操作
    Container,       // 容器隔離——最強隔離，用於不信任的倉庫
}

AppPaths：統一路徑管理

AppPaths 集中管理所有設定檔的路徑，避免路徑邏輯散落在各個模組中：

路徑結構
~/.gcd/                   # 全域設定首頁
    ├── trusted-folders.txt        # 信任規則資料庫
    ├── providers.conf             # 全域供應商設定
    ├── active-provider.txt        # 全域啟用供應商
    ├── checkpoints/               # 檢查點存檔
    ├── sessions/                  # 會話持久化
    ├── commands/                  # 全域自定義命令
    └── skills/                    # 全域技能包

RuntimePreferences：環境變數驅動

執行時偏好完全由環境變數控制，無需設定檔：

環境變數預設值說明
GCD_PROVIDER自動偵測預設供應商 ID
GCD_SANDBOXworkspace-write預設沙箱策略
GCD_TRUSTtrue是否啟用信任檢查
自動偵測邏輯

如果沒有設定 GCD_PROVIDER，系統會依序檢查 CLAUDE_CODE_USE_OPENAI、OPENAI_API_KEY、GEMINI_API_KEY、ANTHROPIC_API_KEY 等環境變數，自動選擇對應的供應商。這讓「零設定啟動」成為可能。

第五章

信任模型 (Trust System)

程式碼位置：trust.rs（168 行）——決定代理能對工作區做什麼。

為什麼需要信任模型？

想像這個情境：你 clone 了一個陌生的開源倉庫，裡面的 .gcd/commands/ 目錄有一個惡意命令會刪除你的 home 目錄。如果代理盲目載入這些命令，後果不堪設想。

信任模型的核心功能是：根據工作區的信任等級，決定是否載入專案層級的指令、命令和技能。

三種信任等級

Trusted

完全信任

載入所有專案指令、命令、技能。適用於你自己的專案。

Parent Trusted

父目錄信任

繼承父目錄的信任。例如你信任 ~/projects/，那麼 ~/projects/any-repo/ 也被信任。

Untrusted

不信任

拒絕載入專案層級設定。只使用全域設定和內建供應商。

最長路徑匹配演算法

當多條規則同時匹配時，GemiClawdex 使用最長路徑匹配——路徑元件數最多的規則勝出：

trust.rs — evaluate()
pub fn evaluate(&self, path: &Path, trust_enabled: bool) -> TrustState {
    // 如果信任機制被停用，直接回傳 disabled
    if !trust_enabled {
        return TrustState { kind: None, matched_path: None, trust_enabled };
    }
    // 找出路徑元件數最多的匹配規則
    let mut best_match: Option<&TrustRule> = None;
    for rule in &self.rules {
        if path.starts_with(&rule.path) {
            match best_match {
                Some(existing) => {
                    if rule.path.components().count()
                       > existing.path.components().count() {
                        best_match = Some(rule);
                    }
                }
                None => best_match = Some(rule),
            }
        }
    }
    // ...
}

信任檔案格式

規則儲存在 ~/.gcd/trusted-folders.txt，格式極為簡單——TSV（Tab-Separated Values）：

trusted-folders.txt
trusted	/home/user/my-projects
parent	/home/user/work
untrusted	/tmp/sketchy-repos/malware-example
安全啟示

信任模型是 GemiClawdex 從 Claude Code 借鏡的最重要設計之一。它示範了一個原則：在 AI 代理中，「能力」必須受「授權」約束。代理能讀寫檔案、執行 shell 命令，這些能力如果不受控，就是安全漏洞。

第十二章

提示詞組裝引擎 — 逐行原始碼導讀

程式碼位置：prompt.rs（615 行）——將 Context、Sandbox、Trust、Files 組裝成最終送給 LLM 的完整提示詞。

為什麼要有 Prompt Engine？

在大多數 AI 代理框架中，「組裝提示詞」和「呼叫 API」是緊密糾纏在一起的。GemiClawdex 刻意將兩者完全解耦——你可以在不連網的情況下測試整個提示詞管線。這就是 Harness Engineering 的核心精神。

1. PromptAssembly 結構宣告

最終的系統提示詞不只是一個字串，而是一個結構化的 PromptAssembly 產物，同時記錄了附件清單與待審批指令：

prompt.rs — PromptAssembly 結構

    pub provider: ProviderProfile,       // 當前選用的供應商與模型
    pub workspace_root: PathBuf,         // 工作區絕對路徑

    pub sandbox: SandboxPolicy,          // 沙箱等級
    pub active_command: Option<String>,  // 觸發的自訂命令名稱 (若有)
    pub active_skill: Option<String>,    // 啟用的技能名稱 (若有)
    pub attachments: Vec<PromptAttachment>, // 解析出來的檔案 / 目錄清單

設計亮點

attachments 欄位記錄了所有被注入的檔案，這讓下游的 Provider 適配器能夠判斷哪些附件需要轉為 Base64 多模態內容 (例如圖片)，哪些只是純文字。

2. assemble_prompt()：四階段管線

assemble_prompt() 是整個引擎的入口函數。它接收一個 PromptRequest，經歷四個階段的轉換後，輸出結構化的 PromptAssembly。

階段 1：命令 (Command) 解析

如果使用者輸入以 / 開頭（例如 /review src/main.rs），系統會將它解析為自訂命令調用：

階段 2：檔案注入 @{...} 與截斷防護

使用者可以寫 @{src/agent.rs}，引擎會自動把檔案內容塞進提示詞中。為了防止 Context Window 超載，GemiClawdex 設定了嚴格的截斷防護：

prompt.rs — 截斷防護常數 + read_text_preview()
const TEXT_INJECTION_LINE_LIMIT: usize = 400;    // 最多注入 400 行
const TEXT_INJECTION_BYTE_LIMIT: usize = 32 * 1024; // 最多注入 32KB

fn read_text_preview(path: &Path) -> io::Result<String> {
    let bytes = fs::read(path)?;
    if looks_binary(&bytes) {
        return Err(/* 擋下二進位檔 */);
    }

    let text = String::from_utf8_lossy(&bytes);
    let mut out = String::new();
    let mut total_bytes = 0usize;

    // 注入時自動帶行號，方便 LLM 精準定位要編輯的位置！
    for (index, line) in text.lines()
        .take(TEXT_INJECTION_LINE_LIMIT).enumerate()
    {
        total_bytes += line.len();
        if total_bytes > TEXT_INJECTION_BYTE_LIMIT { break; }
        out.push_str(&format!("{}\t{}\n", index + 1, line));
    }
    Ok(out.trim_end().to_string())
}
二進位擋截演算

looks_binary() 函數取前 1024 bytes 樣本，如果包含 \0 就直接判定為二進位。若非 null 的控制字元 (byte < 7 或 14-31) 超過 10%，也判定為二進位。這避免了 AI 誤讀圖檔或編譯產物。

第十三章

Agent Loop 狀態機 — 代理的靈魂

程式碼位置：agent.rs（396 行）——prompt → API → tool_calls → execute → loop。這是一切自主行為發生的引擎。

什麼讓 Agent 成為 Agent？

一個普通的 CLI 工具只做一件事就退出。但 AI Agent 有一個迴圈 (Loop)——它會反覆向 LLM 發送請求、解析工具呼叫、在本地執行、把結果餵回去，直到 LLM 判斷任務完成。

1. run_agent()：神聖迴圈完整拆解

這是整套產品的核心函數，位於 agent.rs 第 89 行。我們來逐步拆解：

agent.rs 第 89-148 行 — Agent 思維迴圈
pub async fn run_agent(options: AgentRunOptions)
    -> anyhow::Result<AgentRunResult>
{
    // ① 初始化工具箱與安全上下文
    let tools = builtin_tools();
    let tool_ctx = ToolContext {
        workspace_root: options.workspace_root.clone(),
        sandbox: options.sandbox.clone(),
    };

    // ② 用 Vec 模擬對話歷史 — Agent 的短期記憶
    let mut messages = vec![AgentMessage {
        role: "user".to_string(),
        content: options.initial_prompt.clone(),
        tool_calls: None,
        tool_call_id: None,
    }];

    let mut tool_records = Vec::new();
    let mut final_response = String::new();

    // ③ 啟動回合系統，防止無限遞迴 (預設上限 10 回合)
    for turn in 0..options.max_turns {

        // 步驟 A：呼叫大模型 API
        let response = call_provider_api(
            &options, &messages, &tools
        ).await?;

        // 步驟 B：LLM 是否決定要呼叫工具？
        if let Some(ref calls) = response.tool_calls {
            messages.push(response.clone());

            for call in calls {
                // 步驟 C：在本地執行工具 (帶 Sandbox 控制)
                let result = execute_tool_call(
                    call, &tools, &tool_ctx
                ).await;

                tool_records.push(ToolInvocationRecord {
                    turn,
                    tool_name: call.name.clone(),
                    arguments: call.arguments.clone(),
                    result: result.clone(),
                });

                // 步驟 D：將工具結果掛載為 message 餵回 LLM
                messages.push(AgentMessage {
                    role: "tool".to_string(),
                    content: serde_json::to_string(&result)
                        .unwrap_or_default(),
                    tool_calls: None,
                    tool_call_id: Some(call.id.clone()),
                });
            }
            // → 進入下一輪 for 迴圈
        } else {
            // 沒有工具呼叫 = LLM 認為任務完成
            final_response = response.content.clone();
            messages.push(response);
            break;
        }
    }

    Ok(AgentRunResult {
        turns_used: tool_records.len().max(1),
        final_response,
        messages,
        tool_invocations: tool_records,
    })
}
迴圈的終止條件

Agent 迴圈有兩個退出條件：(1) LLM 回傳純文字、不請求工具 → break；(2) 回合數達上限 max_turns → 自動跳出。這防止了惡意或失控的 LLM 無限循環呼叫工具。

async fn call_provider_api(
    options: &AgentRunOptions,
    messages: &[AgentMessage],
    tools: &[Box<dyn Tool>],
) -> anyhow::Result<AgentMessage> {
    match options.provider.family {
        ProviderFamily::Gemini      => call_gemini_api(options, messages, tools).await,
        ProviderFamily::OpenAiCompatible => call_openai_api(options, messages, tools).await,
        ProviderFamily::Anthropic   => call_anthropic_api(options, messages, tools).await,
    }
}

Gemini 的 Tool Calling 協議不同於 OpenAI。最關鍵的差異是——工具執行結果必須用 "user" 角色傳回：

agent.rs — Gemini 角色映射 (第 186-200 行)

Gemini 回應中的 Tool Calls 也以獨特的 functionCall 格式出現，解析器需要遍歷 parts 陣列分離文字與函數呼叫：

agent.rs — parse_gemini_response() (第 237 行)
fn parse_gemini_response(resp: &Value) -> anyhow::Result<AgentMessage> {
    let parts = resp["candidates"][0]["content"]["parts"]
        .as_array()
        .ok_or_else(|| anyhow::anyhow!("No parts in response"))?;

    let mut text_parts = Vec::new();
    let mut tool_calls = Vec::new();

    for (idx, part) in parts.iter().enumerate() {
        // 文字部分
        if let Some(text) = part["text"].as_str() {
            text_parts.push(text.to_string());
        }
        // 函數呼叫部分
        if let Some(fc) = part.get("functionCall") {
            let name = fc["name"].as_str().unwrap_or("").to_string();
            let args = fc.get("args").cloned()
                .unwrap_or(Value::Object(Default::default()));
            tool_calls.push(ToolCall {
                id: format!("call_{}", idx),
                name,
                arguments: args,
            });
        }
    }

    Ok(AgentMessage {
        role: "assistant".to_string(),
        content: text_parts.join(""),
        tool_calls: if tool_calls.is_empty() {
            None
        } else {
            Some(tool_calls)
        },
        tool_call_id: None,
    })
}

4. 工具分派 execute_tool_call()

每一個工具的呼叫都是安全的——失敗不會讓 Agent 崩潰，而是把錯誤訊息吐回給 LLM，讓它自己除錯：

agent.rs 第 381 行
async fn execute_tool_call(
    call: &ToolCall,
    tools: &[Box<dyn Tool>],
    ctx: &ToolContext,
) -> Value {
    let tool = tools.iter().find(|t| t.name() == call.name);
    match tool {
        Some(t) => match t.execute(call.arguments.clone(), ctx).await {
            Ok(result) => result,
            Err(e) => serde_json::json!({ "error": e.to_string() }),
            //          ↑ 工具失敗 → 把 error 吐回給 AI 就好
        },
        None => serde_json::json!({
            "error": format!("Unknown tool: {}", call.name)
        }),
    }
}
Harness Engineering 設計原則

注意 execute_tool_call 如何讓工具的錯誤也回歸到 LLM 上下文中。這讓 AI 有機會「自我修復」——例如當 write_file 被沙箱阻擋時，LLM 能看到錯誤訊息並換一種策略。

1. Tool Trait 介面設計

在 Rust 中，我們利用 async_trait 設計出簡潔有力的 Tool 介面。如果你要為 AI 新增一種超能力（例如「搜尋網頁」或「執行 SQL」），只需要實作這個 Trait：

tools.rs 第 20-34 行 — Tool trait
#[async_trait::async_trait]
pub trait Tool: Send + Sync {
    /// 機器可讀名稱，用於 function-call 的 payload
    fn name(&self) -> &str;

    /// 人類可讀的一行描述，會被塞進系統提示詞
    fn description(&self) -> &str;

    /// JSON Schema，告訴 LLM 要傳什麼參數進來
    fn parameters_schema(&self) -> Value;

    /// 真正在本地端執行的邏輯！擁有 ToolContext 做安全控制
    async fn execute(
        &self, params: Value, ctx: &ToolContext
    ) -> anyhow::Result<Value>;
}
Trait Object 動態分派

builtin_tools() 回傳 Vec<Box<dyn Tool>>，這是 Rust 的 trait object 模式。它讓我們在執行期間可以動態選擇工具，而不需要用泛型把整個 Agent Loop 的型別搞得非常複雜。

2. 邊界檢查 ToolContext::is_within_workspace()

AI 不可以隨便碰使用者的 /etc/ 或 ~/.ssh/。ToolContext 實作了路徑邊界守衛：

tools.rs 第 43-53 行 — 路徑安全守衛
impl ToolContext {
    pub fn is_within_workspace(&self, path: &Path) -> bool {
        // canonicalize 會解析 ../ 與 symlink，防止路徑穿越攻擊
        let Ok(canonical) = fs::canonicalize(path) else {
            return false;  // 路徑不存在 → 直接擋下
        };
        let Ok(root) = fs::canonicalize(&self.workspace_root) else {
            return false;
        };
        // 嚴格判斷：正規化後的路徑是否以 workspace root 開頭
        canonical.starts_with(root)
    }

    pub fn resolve_path(&self, raw: &str) -> PathBuf {
        let p = PathBuf::from(raw);
        if p.is_absolute() { p }
        else { self.workspace_root.join(p) }
        // 相對路徑一律基於 workspace root 展開
    }
}
防禦 Symlink 穿越攻擊

如果攻擊者在工作區內放置一個指向 /etc/shadow 的 symlink，canonicalize() 會解析成真實絕對路徑，然後被 starts_with(root) 擋下。這是檔案系統安全的基本功。

3. WriteFileTool：層層防護網展示

讓我們拆解 write_file 工具的完整執行路徑——從 LLM 發出寫入請求到檔案真正落盤：

tools.rs — WriteFileTool::execute() (第 132 行)
async fn execute(&self, params: Value, ctx: &ToolContext)
    -> anyhow::Result<Value>
{
    // 防護 1：Sandbox 等級檢查
    if matches!(ctx.sandbox, SandboxPolicy::ReadOnly) {
        anyhow::bail!("write_file is blocked by read-only sandbox policy");
    }

    let path_str = params["path"].as_str().unwrap_or("");
    let content  = params["content"].as_str().unwrap_or("");
    let path = ctx.resolve_path(path_str);

    // 防護 2：路徑邊界檢查 (除非沙箱完全關閉)
    if !matches!(ctx.sandbox, SandboxPolicy::Off)
        && !ctx.is_within_workspace(&path)
    {
        anyhow::bail!("Path is outside the workspace: {}", path.display());
    }

    // 安全通過 — 執行寫入
    if let Some(parent) = path.parent() {
        fs::create_dir_all(parent)?;  // 自動建立中間資料夾
    }
    fs::write(&path, content)?;

    Ok(serde_json::json!({
        "path": path.display().to_string(),
        "bytes_written": content.len()
    }))
}

4. ShellTool：最高權限動作

shell 是權限最高的工具——它能執行任意 Shell 命令。因此受到最嚴格的管控：

tools.rs — ShellTool::execute() (第 222 行)
async fn execute(&self, params: Value, ctx: &ToolContext)
    -> anyhow::Result<Value>
{
    // 被 ReadOnly 與 Container 兩種沙箱模式封鎖
    if matches!(ctx.sandbox,
        SandboxPolicy::ReadOnly | SandboxPolicy::Container)
    {
        anyhow::bail!("shell is blocked by sandbox policy: {}",
            ctx.sandbox.as_str());
    }

    let cmd_str = params["command"].as_str().unwrap_or("");

    // 使用 OS 原生 sh，並鎖定工作目錄為 workspace root
    let output = Command::new("sh")
        .arg("-c")
        .arg(cmd_str)
        .current_dir(&ctx.workspace_root) // ← 不允許逃逸
        .output()?;

    Ok(serde_json::json!({
        "exit_code": output.status.code().unwrap_or(-1),
        "stdout": String::from_utf8_lossy(&output.stdout).to_string(),
        "stderr": String::from_utf8_lossy(&output.stderr).to_string(),
    }))
}

第十五章

會話持久化 (Session Persistence)

程式碼位置：session.rs（646 行）——讓代理擁有「記憶」。

為什麼需要 Session？

沒有 Session，每次執行 gcd exec 都是從零開始。Session 讓你可以：

  • Resume：繼續上次的工作，LLM 能看到之前的上下文摘要
  • Fork：從某個會話分支出去，嘗試不同方向
  • Review：回顧歷史會話，了解代理做過什麼

儲存格式

Session 目錄結構
~/.gcd/sessions/
    └── 1712345678-review/
        ├── session.txt         # 元資料（id, workspace, timestamps）
        └── turns/
            ├── 0001.txt        # 第一輪（含 header + input + prompt）
            ├── 0002.txt        # 第二輪
            └── 0003.txt        # 第三輪

Session ID 生成

ID 格式為 {毫秒時間戳}-{slugified 任務名}，例如 1712345678000-review-providers。slug 化規則：小寫、非字母數字字元轉為 -、去除首尾 -。

Fork 語意

Fork 會複製所有已有的 turn，建立新的 Session ID，並記錄 parent_id 形成譜系（lineage）。這讓你能追蹤一個會話是從哪裡分支出來的。

Session Context 注入

Resume/Fork 時，系統會呼叫 render_session_context() 生成一段摘要，注入到最終提示詞的 # Task 區塊之前。這段摘要包含會話 ID、模式、最近 3 輪的概要，讓 LLM 快速掌握之前的工作。

第十九章

Harness Engineering 原理總結

從 GemiClawdex 的實作中提煉出的 Harness Engineering 設計原則。

七大設計原則

1

分層可組合

指令、命令、技能是三個獨立的層，可以自由組合。這避免了「巨型提示詞模板」的脆弱性。

2

安全即架構

Trust 和 Sandbox 不是外掛的安全模組，而是貫穿整個系統的一等公民。每個模組都知道信任狀態。

3

組裝與執行分離

Prompt Assembly 完全不依賴網路。你可以在離線狀態下測試整個提示詞管線——這是可測試性的極致追求。

4

供應商無關

Gemini、OpenAI、Anthropic 在統一的 ProviderProfile 抽象下等價。切換供應商不需改動提示詞或工具。

5

純文字即真理

設定檔（TSV 信任、INI 供應商、TOML 命令、Markdown 技能）全是人類可讀的純文字。不需要資料庫或二進位格式。

6

Session 即記憶

Resume/Fork 語意讓代理擁有跨執行的記憶能力，且所有狀態都是純文字檔案，方便版本控制。

7

雙模式輸出

每個命令同時支援人類可讀（文字）和機器可讀（JSON）輸出。Headless/CI 與互動使用共用同一套程式碼。

帶走的啟示

如果你只記住一件事

AI 代理的品質，80% 由 Harness 決定，20% 由模型決定。GemiClawdex 用 ~4,500 行 Rust 示範了這一點：模型只是 agent.rs 中的一個 HTTP 呼叫，但圍繞它的工作區發現、信任邊界、沙箱策略、多供應商管理、提示詞組裝、會話持久化等工程——這才是讓 AI 代理真正可靠、可用、可信的關鍵。
