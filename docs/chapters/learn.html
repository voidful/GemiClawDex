這一章的目標

把「看完網站覺得有道理」變成「我知道下一步該去哪裡看原始碼、該問哪些問題、該怎麼練習」。如果前面幾章是在建立架構感，這一章就是把那份架構感變成可操作的學習路徑。

三條閱讀路徑

3 分鐘

完全新手先上手

依序看 首頁 → 前言 → 入門章。目標只有一個：先知道 agent 比聊天模型多了哪幾層，不求一次記住所有術語。

6 分鐘

把術語先分開

依序看 首頁 → 前言 → 入門章 → 術語速查。如果你會把 trust、permission、session、memory、provider 混在一起，先走這條會更順。

5 分鐘

先建立 外殼 基本概念

依序看 首頁 → 前言 → 入門章 → 術語速查 → GCD 全景。目標不是記住所有模組，而是先理解「LLM 只是引擎，外殼才是產品」。

15 分鐘

完整走一遍所有章節

從前言開始，依序走過入門 → 架構 → 流程 → 各個細節章節 → 比較與參考。用大約 15-30 分鐘，就能對整個系統有完整的認識。

30 分鐘

帶著工程判斷讀完

全部章節走一遍，再回到原始碼對照。這條路適合已經有一定程式開發經驗的讀者。

不同背景的讀者，應該各自抓什麼重點

你目前的背景 建議先看 先不要急著鑽哪裡
完全沒做過 agent 前言、入門章、術語速查、GCD 全景、Pipeline 不要一開始就執著在 provider API 細節，先把 外殼 六層架起來。
做過 prompt chaining / tool calling Pipeline、對照章、參考矩陣 不要只把 GCD 看成 prompt template 系統，真正的重點在 trust、permission 與 session。
在做本地 AI 助手 或 CLI GCD 全景、Pipeline、Roadmap 不要被功能名稱帶走，先確認哪些邊界已加入主要流程。
想把專案當教材 前言、全景、對照章、參考附錄 不要先從產品比較開始，否則讀者很容易只記得品牌差異，忘了 外殼 結構。
想為自己的專案寫 harness Context Engineering → AGENTS.md 工作坊 → 驗證迴圈 不要跳過 Context Engineering。理解 context 組裝邏輯，是寫好 AGENTS.md 的前提。

從網站回到原始碼的五個入口

1

crates/gcd-core/src/app.rs

先看 exec()。這裡最能看出一個 外殼 會先做哪些準備工作，哪些層是 執行前的準備。

2

crates/gcd-core/src/prompt.rs

接著看提示詞組裝。理解提示詞怎麼被組合出來，比先看 AI 模型 API 更能讓你理解 AI 助手怎麼運作。

3

crates/gcd-core/src/agent.rs

再看工具迴圈、權限控制與備援切換。這是 GCD 的執行核心。

4

crates/gcd-core/src/tools/

看沙盒怎麼在不同工具上實際生效。可以幫你判斷「宣稱的安全」和「實際的安全」有沒有差距。

5

crates/gcd-core/src/session/

最後看 session 持久化與 replay。這能幫你把 agent 看成持續工作系統，而不是一次性 prompt。

帶著問題走回原始碼，比從檔名亂逛有效

看 app.rs 時問：哪些事發生在模型呼叫之前？

如果一個能力在 App::exec 之前就決定，通常代表它屬於 外殼 的執行前的設定，而不是模型推理的一部分。

看 prompt.rs 時問：最終 prompt 的每一段從哪裡來？

不要只看字串內容，更要看來源：repo instructions、commands、skills、attachments、工作接續 各自負責解哪一種資訊不足的問題。

看 agent.rs 時問：tool call 和 操作影響 中間隔了哪些 gate？

這個問題可以幫你快速看出一個 agent 的安全設計是實作還是空談。

看 session/ 時問：這些資料會不會回到下一輪 prompt？

如果不會，它就比較像 logging；如果會，它才開始接近真正的 工作接續。

推薦練習

練習 1：追一次 /review 的 prompt 展開

從 commands.rs 找命令，再到 prompt/command_invocation.rs 看 {{args}} 如何進 最終提示詞。你會清楚感受到 workflow file 本身也是 外殼 的一部分。

練習 2：改變 trust 狀態，觀察 instructions 是否消失

對照 trust.rs 與 instructions.rs，理解「trusted folder」不是 UI 標籤，而是直接影響 提示詞組裝 的 gate。

練習 3：沿著一個 tool call 往回看

從 execute_tool_call 往上看 adapter 解析，再往下看 ToolContext 和具體工具。這能讓你把模型請求與真實系統 操作影響 串起來。

練習 4：對照文件的 Implemented 標記

挑一個像 MEMORY.md 或 container sandbox 的項目，確認它是不是只有局部連接。這個練習能訓練你避免被 AI 助手專案的模組名稱誤導。

練習 5：追一次 hook deny 與 sub-agent 委派

先看 hooks.rs 的退出碼含義，再看 tools/coordinator.rs 如何建立 sub-agent options。這能幫你理解 安全管理層不只是 prompt 或工具清單，而是另一套實際插點。

練習 6：把同一條規則分到 instructions / command / skill

挑一個常見任務，例如 review 或 bugfix，試著自己判斷哪些內容應該常駐、哪些應該命令化、哪些應該做成方法論 skill。這能訓練你設計工作流語言，而不是只會堆 prompt。

練習 7：自己解釋一次 trust / permission / sandbox 的差別

不要先背定義，試著用「它在問什麼問題」來解釋這三層。能自己講得清楚，後面讀工具與 trust 章時就比較不會混淆。

如果你不知道怎麼開始，這裡有一個 20 分鐘實作練習

01

先假設自己是 runtime

拿一個任務，例如「review src/app.rs」，先在紙上寫出：哪些上下文要在模型呼叫前決定？哪些是 prompt 組裝時才知道？哪些要靠工具？

02

再對照 GCD 主要流程

回到 app.rs、prompt.rs、agent.rs，逐一確認你的猜測是否真的在程式裡出現。這一步能幫你把抽象架構轉成執行順序。

03

最後試著自己判斷完成度

挑一個功能（例如 memory、MCP、container），試著自己判斷它是完整實作、部分實作、還是只有基本架構，再回來對照網站的說法。

初學者可直接複製的觀察筆記模板 
任務：
我要理解 /review src/app.rs 這件事是怎麼完成的

我先猜：
1. 工作區與 trust 會先決定
2. prompt 會載入 AGENTS.md
3. 模型可能呼叫 read_file / shell
4. 結果應該會寫進 session

我在原始碼看到：
- app.rs: 先 discover workspace，再 load trust / providers / commands
- prompt.rs: 組 最終提示詞，插入 instructions 與 待審核指令s
- agent.rs: 解析 tool call，經過 permission / hooks
- session/: turn 與 events 會被保存

我的判斷：
- 提示詞組裝 = Implemented
- memory integration = Implemented

如果你要拿這個網站做讀書會或內部教學

01

第一段先建立結構，不先比產品

用前言與全景章把六層 外殼 建起來。這一步的成果是讓所有人先有共同詞彙。

02

第二段再看一條真正會動的 pipeline

把 trust、sandbox、permission、提示詞組裝 與 session 放在同一張圖裡看，讀者才會理解這些層不是平行 feature，而是執行順序上的關卡。

03

第三段最後才做三大產品對照

等讀者已經知道 外殼 結構後，再講 Claude Code、Gemini CLI、OpenAI Codex 的差異，吸收速度會快很多，也比較不會變成品牌印象比較。
