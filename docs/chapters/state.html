工作記錄和記憶

這三個詞在 AI 助手文件裡最容易被混成一團。這一章專門講清楚：什麼是執行紀錄、什麼是下一輪上下文、什麼又只是尚未完全連接的記憶介面。

先講結論

在 GCD 裡，session 是正式的執行記錄與延續基礎；continuation 是從既有 session 中抽出的摘要，重新插回下一輪 prompt；memory 則是雙重記憶系統（MEMORY.md + USER.md），透過 memory 工具讓 agent 自主讀寫，每次執行時以 fenced block 注入 prompt。另外，session_search 工具讓 agent 能搜尋過往 session 紀錄，回溯類似問題的解法。這四者不能互相替代。

為什麼這一層值得單獨分開講

很多文件會把「記憶」「會話」「上下文延續」通通包在同一段描述裡，結果讀者只知道系統似乎有在保存東西，卻不知道保存的是什麼、下一輪到底會不會用到、以及哪些只是未來規劃。對 GCD 來說，這一點特別重要，因為它直接決定系統是不是只會做一次性 prompt，還是能支援 resume、fork、replay 與可回放教學。

概念 它回答的問題 在 GCD 的現況
Session 這次執行發生了什麼？之後可不可以再看、再接、再分叉？
Continuation 既有 session 的哪些內容要回到下一輪 prompt？
Memory 跨 session 的持久知識要怎麼獨立保存與回用？

Session 在 GCD 裡不是「聊天記錄」，而是執行紀錄

GCD 的 session 並不只存使用者輸入與最終回答。每一個 turn 都會保存 provider、sandbox、trust、active command、active skill、完整 prompt、final response、tool 次數、token 用量、API calls 與 events。這個資料量，讓 session 更接近執行紀錄而不是對話紀錄。

S1

保存 prompt 本體

這代表你之後可以回頭看「當時到底送了什麼 prompt」，而不是只剩模糊印象。

S2

保存事件流

ExecutionPrepared、ToolCallRequested、ProviderFallback 等事件都能被 replay。

S3

保存執行條件

provider、sandbox、trust、permission 模式的差異都會直接改變結果，因此必須跟著 turn 一起保存。

S4

支援 resume / fork / replay

這使 session 不只是存檔，而是後續 workflow 的基礎設施。

GCD 的 session 檔案實際上保存了什麼

session/storage.rs 會把 session metadata 寫進 session.txt，再把每個 turn 寫成獨立檔案。turn 檔內含 header 與 payload，header 記錄位元組長度，payload 再依序拼在一起 input、prompt、response 與 events JSON。這個設計很樸素，但也很誠實：它明確表示 session 的核心材料是什麼。

範例：一個 session turn 檔案大概會長什麼樣子 
index=1
timestamp_secs=1712345678
mode=live
provider_id=openai-codex
provider_label=OpenAI Codex
sandbox=workspace-write
trust_label=trusted
active_command=review
input_bytes=26
prompt_bytes=1420
response_bytes=320
events_json_bytes=980
turns_used=2
tool_invocation_count=3
prompt_tokens=1200
completion_tokens=220
total_tokens=1420
api_calls=2

[input][prompt][final_response][events_json]

這裡很值得注意的一點是：完整 prompt 也被保存了。這讓 session 成為一個很強的學習資源，因為你不只知道 agent 做了什麼，還知道它是在什麼上下文下做出這些決策。

resume、fork、replay 三者有什麼差別

模式 它要解的問題 在 GCD 裡的行為
resume 我想沿著同一條工作繼續做下去。 沿用原 session id，並在下一個 turn 追加新紀錄。
fork 我想從既有工作分叉出另一條嘗試。 複製既有 turns，建立新 session id，並記錄 parent session。
replay 我想回頭看某次執行到底發生什麼。 把保存的 events 重新列出，讓除錯與教學都可追蹤。
範例：同一段工作在 resume 與 fork 下的差異 
你完成了一輪 /review src/app.rs

resume
=> 同一個 session 繼續新增 turn 2
=> 適合「延續剛才那件事」

fork
=> 複製原 session，建立新 session id
=> 適合「保留原路線，再試另一種修法」

重點：
兩者都建立在 session 存得夠完整，
否則分叉與延續都只會變成薄薄的聊天歷史。

Continuation 不是全部重灌歷史，而是濃縮回傳

當使用者選擇 resume 或 fork 時，GCD 不會把全部過去訊息生硬地重新塞進 prompt，而是透過 render_session_context() 產生一段 # Session Continuation 區塊，摘要 session id、工作區、parent session 與最近三個 turn 的概況。這種設計很務實，因為它在保留脈絡與控制 提示詞變太大之間做了折衷。

範例：continuation 區塊的形狀 
# Session Continuation
Mode: resume
Session ID: 1712345678-review-src-app-rs
Workspace: /workspace/project
Existing turns: 3
Parent session: 1712341000-review-src-app-rs
Recent turn summaries
- [1] /review src/app.rs | mode=live | provider=openai-codex | sandbox=workspace-write | tools=2
- [2] fix error handling in src/app.rs | mode=live | provider=openai-codex | sandbox=workspace-write | tools=3
- [3] re-run tests and summarize remaining risks | mode=live | provider=gemini-official | sandbox=workspace-write | tools=1

這裡也能看出一個很重要的區分：continuation 是從 session 中抽出來的可用摘要，不是另一套獨立資料系統。如果 session 本身存得不完整，continuation 也不可能憑空變得有用。

那 memory 呢？為什麼它不能跟 session 混成同一件事

參考 hermes-agent 的設計，GCD 已定義雙重記憶系統，並透過 memory 內建工具讓 agent 在執行中自主讀寫：

  • agent store（MEMORY.md）：環境觀察、專案慣例、工具怪癖。存放在工作區 .gcd/MEMORY.md。
  • user store（USER.md）：使用者偏好、溝通風格、工作流習慣。存放在工作區 .gcd/USER.md。

每個 entry 用 §（section sign）分隔，支援多行內容。App::exec 會先嘗試新的雙重記憶格式，如果不存在則 fallback 到舊的單一 read_memory 格式。注入 prompt 時使用 <memory-context> fenced block，防止模型將回憶內容當成新的使用者指令。

如果你想保存的是 比較適合放在哪裡 原因
某次執行的完整 prompt、tool calls、tokens、events Session 這是執行紀錄，未來要 replay / resume / fork 都會用到。
跨多次執行都值得保留的專案知識或工具怪癖 Memory（agent store） 透過 memory 工具寫入 MEMORY.md，每次執行自動注入 prompt。
使用者個人偏好與工作習慣 Memory（user store） 透過 memory 工具寫入 USER.md，跟 MEMORY.md 一起注入 prompt。
為了下一輪工作只需要一小段歷史摘要 Continuation 它的目標是控制 prompt 尺寸，同時保留足夠上下文。

實況檢查：memory 的實作現況

雙重 store 已完整可用

memory 工具支援 add、replace、remove、read 四個 action，分別操作 agent（MEMORY.md）與 user（USER.md）兩個 store。

已接入提示詞組裝主要流程

App::exec 先嘗試 build_memory_prompt_block（雙重記憶格式），fallback 到舊的 read_memory。assemble_prompt 會偵測 <memory-context> tag 並直接注入 fenced block。

安全掃描已內建

寫入前掃描 prompt injection、exfiltration pattern、invisible unicode。每個 entry 上限 4000 chars，每個 store 上限 100 entries / 50K chars。

Session 搜尋已可用

session_search 工具可搜尋 ~/.gcd/sessions/ 中的 JSONL 歷史紀錄，回溯類似問題的解法。屬於輕量檔案搜尋，不依賴 SQLite FTS5。

雙重記憶已完整接入，但仍有潛在改進空間

MEMORY.md + USER.md 的內容現在會自動注入每次執行的 prompt，寫入前有安全掃描。但真正完整的 long-term memory 還可以回答更多問題：何時自動摘要、何時清理過期記憶、怎樣依相關性過濾。這些都是未來可以持續優化的方向。

如果你是貢獻者，該怎麼分辨新資料該進 session 還是 memory

判斷問題 如果答案偏向是 比較適合去哪裡
這是某一輪執行的事後證據嗎？ 是 Session
它應該跨多輪、甚至跨多次 session 持續存在嗎？ 是 Memory
它只是要讓下一輪快速理解目前做到哪裡嗎？ 是 Continuation
它需要支援 replay / fork / audit 嗎？ 是 Session
