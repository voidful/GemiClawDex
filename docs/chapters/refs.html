參考來源

這裡不是單純堆連結，而是把每個來源對本站說明與 GCD 判斷的影響範圍寫清楚。

可追蹤原則

如果某個判斷直接來自 GCD 原始碼，就標示原始碼依據。如果來自外部資料，就標明來源與用途。如果是基於社群分析或文件總結，會特別標示，避免讓讀者誤以為是專案官方聲明。

原始碼錨點：寫文件時最常對照的檔案

原始碼檔案 這個檔案告訴我們什麼
app.rs 整個系統的入口。工作區偵測、信任載入、provider 設定都在這裡，可以判斷哪些功能真的在執行流程裡。
prompt.rs 提示詞組裝的核心。看 commands、skills、附件、待審核指令怎麼被合併成最終提示詞。
agent.rs AI 對話迴圈的核心。包含 provider 呼叫、工具迴圈、權限檢查、hooks、備援切換。
hooks.rs PreToolUse / PostToolUse 的實作。看 hook 怎麼攔截工具呼叫，退出碼各代表什麼。
tools/coordinator.rs Sub-agent（spawn_agent）的實作。包含回合上限、深度上限、繼承條件。
trust.rs 信任判斷規則。決定工作區是 trusted / parent-trusted / untrusted，直接影響哪些設定會被載入。
instructions.rs 專案規範載入。只有可信工作區才會讀 AGENTS.md、GEMINI.md 等設定檔。
commands.rs 自訂命令的載入與 {{args}} 置換。任務模板怎麼變成提示詞的一部分。
skills.rs 技能包的 Markdown 載入、名稱推導、顯式選用。技能是方法論知識包，不是命令模板。
tools/shell.rs
tools/container.rs		 Container 沙盒怎麼對 shell 工具提供 Docker 隔離執行。
session/storage.rs
session/render.rs		 工作記錄的保存與回放。checkpoint、resume、fork、replay 的實作都在這裡。
mcp.rs MCP 伺服器初始化、工具發現與 JSON-RPC 封裝。外部工具擴充就靠這個。
agent/runtime_support.rs 上下文預算管理、訊息壓縮、自動 git commit 與 commit 訊息清理。
output.rs 輸出格式控制。--json 和 --jsonl 不只是換格式——它們輸出的資料結構不同。
gcd-cli/main.rs CLI 入口。子命令對應、REPL 模式、--plan / --json 等參數的實作。
agent/memory.rs MEMORY.md 的讀寫。已連接到提示詞組裝流程，可以跨工作記錄記住事情。

三個參考型判斷範例

如果你想知道這個附錄到底怎麼被使用，下面三個樣本就是最實際的答案。它們示範了本站如何從檔案與程式路徑，一步步推到判斷。

判讀題目 我們先看哪裡 最後如何落結論
為什麼 MEMORY.md 不是完整記憶系統 agent/memory.rs 是否被 app.rs、prompt.rs、agent.rs 主要流程持續使用。 因為目前只看到讀寫介面，沒有穩定回傳到 prompt 或 工作接續，所以標 。
為什麼 規劃模式 不能被寫成獨立 agent runtime App::exec 中 plan_only 分支是否真正進 run_agent。 它目前更接近規劃輸出模式與 session / checkpoint 記錄，而非獨立的 live planning 工具迴圈。
為什麼 container sandbox 不是完整隔離含義 tools/shell.rs 與其他工具是否都改走 container.rs。 只有 shell 真正切到容器執行，所以文件必須保守標示，不可誤寫成全工具隔離。
為什麼 instructions、commands、skills 不該互相亂塞 instructions.rs 的固定檔名載入、commands.rs 的 slash invocation、skills.rs 的顯式選用，是否真的代表不同觸發含義。 因為這三層處理的是不同粒度的工作知識，文件必須把它們寫成工作流語言，而不是同義 prompt 片段。
為什麼 spawn_agent 不能直接被寫成完整多代理平台 tools/coordinator.rs 是否有限深度 / 回合、是否經過 App::exec、是否保留 plugin / session / fallback 全貌。 因為它目前更接近 有限制的委派 tool，所以標 的工作代理執行面，而不是完整協調層。

如何把附錄當成研究註腳，而不是無用連結清單

當你在主文看到一個判斷時

先回來附錄找對應的原始碼錨點，確認它是被哪一條主要流程支持，而不是只因為存在某個模組名稱。

當你在主文看到一個參考說法時

回來附錄找它是來自產品、理論、從頭重建 還是社群反饋。這會幫你判斷那句話的可信度與用途。

當你想自己做新判讀時

先找原始碼錨點，再找對應的外部座標，最後才下判斷。這也是本站整體分析框架的基本動作。

附錄使用範例：從一句主文判斷回到證據 
主文句子：
"MCP 已加入工具註冊表，但 client 實作較簡化，應保守看待。"

我回附錄會看：
1. mcp.rs 是否真的有 initialize / tools/list / tools/call
2. agent.rs 是否真的把 MCP tools merge 進 all_tools
3. 是否看到更完整 capability negotiation / robust error handling

因此得到：
- 主要流程確實能載入 MCP tools
- client 具備 initialize / tools/list / tools/call 完整流程
- 標成 Implemented

使用外部來源時，本站刻意遵守的誠實界線

原始碼可以證明的，就不借外部話術來包裝

例如 GCD 的 信任檢查、提示詞組裝、工作記錄保存，都優先回到本專案原始碼確認，而不是用外部產品官網的語言來描述。

外部產品提供的是參考方向，不是 GCD 現況

Gemini CLI 的 token caching、Codex 的本地 agent UX、Claude Code 的工作流拆解，都可以成為比照座標，但不會被直接寫成 GCD 已完成。

社群分析只會標成 Implemented

社群還原樹、繁中解說、從頭重建都很有價值，但本站不會把它們包裝成原版官方聲明。

來源越接近 實際使用情況，越值得放進比較章

成本、相容性、本地化、安全透明度這些訊號，雖然未必華麗，卻最能說明為什麼某些設計值得被借進 GCD。
