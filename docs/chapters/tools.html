Tools

工具和權限

AI 模型不會自己讀檔案、跑指令。這一章介紹 GCD 的 11 個內建工具，以及系統怎麼控制這些工具能做什麼。

先講結論

在 AI 助手 裡，工具層不是模型的附屬配件，而是 操作真的發生的地方。GCD 用 Tool trait、ToolContext、sandbox policy、permission level 與 hooks，把「模型提出的意圖」轉成「系統允許的動作」。如果這一層不夠清楚，agent 很快就會從有用工具變成無法控制腳本。

先把工具層想成什麼

一個大型 prompt 即使寫得再好，也不能自己讀檔、不能真的改檔、不能跑 shell、更不會自動知道哪個路徑超出工作區。這些事情都必須套用到工具實作與 執行時 gate。換句話說，工具層其實是在回答一個很實際的問題：模型說想做什麼時，系統真的允許它做到哪裡。

1

工具定義能力面

每個工具都透過 name、description、parameters_schema 對 provider 揭露能力與輸入形狀。

2

ToolContext 定義執行邊界

工作區根目錄、sandbox、hooks、coordinator provider / key / permission 都由 context 提供。

3

Permission 定義人機節奏

同一個 tool call 在 suggest、auto-edit、full-auto 下，允許方式會不同。

4

Hooks 定義外加政策

即使工具本身允許執行，PreToolUse / PostToolUse 仍可以加上額外攔截、紀錄與整合。

GCD 目前有哪些 built-in tools

tools/registry.rs 會先建立 built-in tools，再把 plugin tools 與 MCP tools 併進來。這表示工具目錄不是單一靜態清單，而是一個可擴充的註冊表。

修改 write_file、apply_patch 產生或修改工作區檔案。 高，會直接改變工作區內容。

ToolContext 真正做的是邊界計算

ToolContext 很值得特別看，因為它不是單純把幾個變數打包，而是在工具真正執行前，提供「這次執行允許的空間」。像 workspace_path() 會先 canonicalize 工作區，再解析既有祖先路徑，避免路徑穿越或偷偷跳出工作區。這讓 sandbox 不只是字串標籤，而是進入具體檔案路徑判斷。

範例：workspace_path() 在保護什麼 
模型提出：
write_file(path="../outside.txt", content="...")

ToolContext 會做：
1. 解析工作區根目錄的 canonical path
2. 嘗試解析目標路徑既有祖先
3. 檢查目標是否仍落在 workspace root 之下

結果：
如果路徑跳出工作區，工具直接失敗，不會寫出去

同樣是 sandbox，對不同工具的意義不一樣

很多人會把 sandbox 想成單一保護罩，但對不同工具來說，它的落點其實不同。讀取類工具主要受工作區邊界限制；寫入類工具還會額外受 read-only 限制；shell 則在 container 模式下改走完全不同的執行路徑。

工具 off read-only workspace-write container
read_file / list_dir / search_files 可讀取絕對路徑或相對路徑。 限制在工作區內。 限制在工作區內。 限制在工作區內。
write_file / apply_patch 可寫絕對或相對路徑。 直接阻擋。 允許在工作區內修改。 目前仍直接作用在主機工作區路徑。
shell 在主機上執行。 直接阻擋。 在工作區中執行。 改走 Docker / Podman 容器。
fetch_url 可執行。 目前程式碼未額外阻擋。 可執行。 目前程式碼未額外阻擋。

這裡最值得注意的 實況檢查 有兩個。第一，container 模式目前只完整包住 shell，其他讀寫工具還是在主機工作區上操作。第二，fetch_url 目前沒有依 sandbox policy 額外收斂網路存取。這些都不是要否定設計，而是提醒文件必須忠實反映實作邊界。

權限不是工具能不能用，而是要不要人工批准

PermissionLevel 和 sandbox 常被混在一起，但它們處理的問題不同。sandbox 比較像環境邊界；permission 則在回答「這次動作要不要經過人類點頭」。在 GCD 裡，suggest、auto-edit、full-auto 會對不同工具產生不同要求。

Permission 代表什麼 哪些工具通常需要確認
suggest 偏建議模式，任何真正寫檔或執行命令都更應該問人。 write_file、apply_patch、shell
auto-edit 允許較多自動修改，但仍把高風險工具保留給人工確認。 shell、apply_patch
full-auto 在既有 sandbox 內盡量自動執行。 原則上不需額外確認，但仍受 sandbox 與 hooks 影響。
範例：同一個 patch 在不同 permission 下的命運 
Tool call: apply_patch(path="src/app.rs", patch="...")

suggest
=> 會要求人工確認，否則不執行

auto-edit
=> 仍可能要求確認，因為 patch 屬於高影響工具

full-auto
=> 如果 sandbox 允許，runtime 可直接執行

重點：
permission 解的是「誰拍板」，
不是「底層環境允不允許」

Hooks 是工具層外掛式政策，不是單純日誌

GCD 的 hooks 系統允許在工具前後執行 shell command。PreToolUse 可以阻擋，PostToolUse 則偏向通知與整合。這個設計的好處，是團隊可以不改每個工具的實作，就插入自定政策，例如禁止某些 shell pattern、把工具結果送到外部監控、或在修改後自動跑額外檢查。

H1

exit 2 代表 deny

PreToolUse hook 如果以 exit code 2 結束，該工具呼叫會被拒絕。

H2

工具參數會變成環境變數

GCD_TOOL_NAME、GCD_TOOL_ARGS、GCD_TOOL_RESULT 讓 hook 能讀到上下文。

H3

Hook 有 timeout

預設 10 秒，避免某個 hook 把整條工具鏈卡住。

H4

Workspace hook 目前很值得保守描述

因為 hook 載入路徑與 trust 含義還需要更清楚文件化，這是值得持續 實況檢查 的地方。

範例：一個簡單的 PreToolUse hook 
{
  "pre_tool_use": [
    {
      "command": "echo \"$GCD_TOOL_ARGS\" | grep -q 'rm -rf' && exit 2 || exit 0",
      "tools": ["shell"],
      "timeout_secs": 5
    }
  ]
}

這個例子要表達的不是最佳實踐，而是 hooks 的定位：它們讓團隊可以在工具執行前後加上一層本地政策，而不必每次都回頭改 代理迴圈 核心邏輯。

spawn_agent 代表什麼完成度

CoordinatorTool 讓 GCD 可以把子任務交給 sub-agent。這表示 GCD 已經不只是單線工具執行器，而是開始具備協調多輪工作的能力。不過它現在也有很清楚的限制：最大深度 2、sub-agent 不串流輸出、不 auto-git，而且呼叫 run_agent 時沒有沿用父層 plugin tool list。

面向 目前行為 判讀
遞迴深度 上限 2 避免無限展開，但也表示還不是完整多代理協調系統。
繼承上下文 沿用 workspace、sandbox、provider、permission 這讓子代理不是完全陌生的新工作。
工具集 會得到 built-in tools 與 MCP，但不會直接收到父層 plugin tools 清單 屬於可用但仍有限制的實作。
輸出模式 不串流、不 auto-git 較適合 有限制的委派，不適合拿來模擬完整並行工作平台。

Hermes-inspired 記憶與學習工具

參考 hermes-agent 的設計，GCD 新增了三個讓 agent 從「工具使用者」升級為「自我改進的學習者」的工具。這三者都不操作工作區原始碼，而是操作 ~/.gcd/ 下的持久知識檔案。

H1

memory：雙重記憶

agent store（MEMORY.md）記錄環境觀察與專案慣例；user store（USER.md）記錄使用者偏好與工作習慣。每次執行時自動以 <memory-context> fenced block 注入 prompt，防止模型將回憶當成新指令。

H2

skill_manager：技能學習迴圈

支援 create、edit、patch、delete、write_file、remove_file 六個 action。只能修改 user-created skills（~/.gcd/skills/），bundled 與 project-level skills 為唯讀。

H3

session_search：對話回溯

搜尋 ~/.gcd/sessions/ 中的 JSONL 歷史紀錄，關鍵字全部匹配後回傳 snippet（match position ± 200 chars context），按修改時間排序，預設最多 5 筆結果。

這三個工具的安全機制

memory 與 skill_manager 的所有寫入操作都會先經過安全掃描：檢查 prompt injection pattern（如 ignore previous instructions）、exfiltration pattern（如 curl ... $API_KEY）、以及 invisible unicode characters（zero-width space、directional overrides 等）。掃描不通過就直接拒絕寫入。

實況檢查：工具層哪些地方已經很完整，哪些還不能寫滿

能力 判讀 說明
讀取 / 搜尋 / 寫入 / patch 類工具 主要流程已完整可用，且都經過 ToolContext 與 sandbox 邏輯。
Shell with container path container 模式目前只完整包住 shell。
Hooks lifecycle PreToolUse / PostToolUse 都已在 工具迴圈 前後生效。
Network gating for fetch_url 工具可用，但目前 sandbox policy 沒有對網路 access 做額外收斂含義。
Coordinator / sub-agent 已能工作，但應以 有限制的委派 心智理解，而非完整多代理平台。
雙重記憶系統（Hermes-inspired） MEMORY.md + USER.md 讀寫已接入主要流程，prompt 注入使用 fenced block，寫入前含安全掃描。屬於 Implemented。
