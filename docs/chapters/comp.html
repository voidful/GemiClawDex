對照

Claude Code、Gemini CLI、OpenAI Codex、Hermes Agent 對 GCD 的影響

這一章不做品牌評比，而是明確回答：學了什麼、沒學什麼、為什麼這樣取捨。

GCD 不是三個產品的拼裝版。它更像是一個刻意縮小、用 Rust 重整的 教學範例：保留三大產品最值得學的設計思路，同時避免直接把大型商業系統的歷史複雜度整包搬進來。這一章的重點不是比誰比較強，而是回答一個更實際的問題：哪一層的觀念值得學，哪一層的歷史負擔應該刻意不借。

先講一個總結論

如果把四個主要參考來源的影響分開來看，Claude Code 對 GCD 的影響主要在 prompt、tool、trust 與 workflow 語言；Gemini CLI 主要影響 terminal-first 心智、workspace config 與 MCP / context file 的擴充方向；OpenAI Codex 則提供了 approval、sandbox、本地 agent 交互節奏的產品風格；Hermes Agent 貢獻了技能學習迴圈、記憶系統和 session 搜尋。GCD 把這些脈絡重新編排成一條 Rust-native 主要流程，而不是把四個來源的功能名稱拼成清單。

讀法提醒

下面會出現一些「如果同一個任務放進不同產品脈絡會怎麼看」的範例。這些例子是為了幫助理解設計方向，不代表三家產品的官方逐字行為，也不會拿來當 GCD 已實際做出來功能的證據。

三個主要參考來源

Claude Code

借 prompt / tool / trust 的深度

主要依據社群社群分析、教學拆解與相關繁中文件。GCD 學的是 repo instructions、skills、workflow、信任規則 這些 外殼設計。

  • 學了什麼：AGENTS 類指令檔、技能包、工具導向 workflow、明確 trust 邏輯。
  • 沒學什麼：大規模 大型單體系統、內部遙測與成堆 feature flag、產品級 worktree orchestration。
  • 為什麼：GCD 要保留可讀性與可以在本地測試，不能為了像 Claude Code 而把複雜度抬回去。
Gemini CLI

借 terminal-first 與可擴充性基本概念

的是 terminal-first、workspace config、MCP awareness、provider profiles； 的是 token caching 與更成熟的 擴充生命週期。

  • 學了什麼：trusted folders 心智、context file 思維、headless / JSON output、MCP 生態接點。
  • 沒學什麼：完整 token caching、完整多模態工作流、成熟 extension 平台。
  • 為什麼：GCD 先穩主要流程，再補 cache 與擴充生命週期，避免把整合成本攤得太早。
OpenAI Codex

借 approval / sandbox / local agent UX

的是本地執行節奏、approval / sandbox / permission 語言； 的是更完整的安全隔離體驗。

  • 學了什麼：instruction-driven workflow、審核機制、sandbox 分級、本地 AI 助手 手感。
  • 沒學什麼：單一供應商假設、完整 app shell / GUI、產品級隔離與雲端整合。
  • 為什麼：GCD 想保留 provider 可替換性，並把安全設計寫進原始碼與文件，而不是綁在單一平台。

第四個主要參考：Hermes Agent

為什麼單獨列出

前面三個是商業產品。Hermes Agent 是開源社群專案（NousResearch），但它對 GCD 的影響不亞於三大產品，尤其在「agent 怎麼從經驗中學習」這個方向上。GCD 的技能學習迴圈、雙重記憶系統、記憶安全掃描、Session 搜尋、Progressive Skill Disclosure 五個特性都來自 Hermes。

Hermes Agent

借 skill learning loop / memory / session search

Hermes 是目前最完整的開源 AI agent 之一：205K 行 Python、78 個 skills、26 個類別、支援 6 個通訊平台。GCD 學它的自我改善機制，不學它的平台規模。

  • 學了什麼：skill_manager 自動建立技能、MEMORY.md + USER.md 雙重記憶、寫入前安全掃描、session_search 回溯、YAML frontmatter + progressive disclosure。
  • 沒學什麼：gateway 多平台（Telegram / Discord / Slack）、skin 主題引擎、credential pool、batch trajectory generation、RL 訓練環境。
  • 為什麼：GCD 是教學專案，專注在「一個工程師在終端機前怎麼用 agent」這個場景。多平台支援和 RL 訓練超出教學範圍。

把同一個任務放進三種產品風格裡看

假設目標都是「在本機工作區 review 一個 Rust 檔案並提出修改建議」，三種脈絡最容易讓人感受到的差異如下：

觀察角度 Claude Code 系脈絡 Gemini CLI 系脈絡 OpenAI Codex 系脈絡 GCD 的吸收方式
你最先感受到的是什麼 repo instructions、skills、workflow 語言很重。 terminal-first、context file、trusted folder 與工具擴充感很重。 approval、sandbox、本地 agent 節奏很明顯。 把三者分成不同層，避免全部混成同一種「系統 prompt 魔法」。
同一個 review 任務會被怎麼定義 更像「在既定工作流與規範下完成任務」。 更像「在 terminal 中帶著上下文與外部工具完成任務」。 更像「在可控批准與隔離下完成本地變更」。 把 workflow、工具上下文與 permission 模型都保留，但用 Rust 主要流程重寫。
最值得學給 GCD 的 prompt / skill / trust 的深度。 產品邊界與擴充心智。 人機協作節奏與 approval 語言。 形成 GCD 現在的 提示詞組裝、terminal-first、permission policy。

Claude Code 對 GCD 的影響，不只是「像不像」

GCD 從 Claude Code 系脈絡借得最深的，不是介面或品牌語彙，而是 prompt 與工作流的層次感。Repo instructions、skills、commands、tool-first workflow、以及 trust 對 instructions 載入的影響，這些都讓 GCD 比較接近一個真正的 AI 助手，而不是會寫程式的聊天機器人。

但 GCD 沒有複製 Claude Code 的地方也很明確。它沒有打算把大型內部產品多年堆出的歷史結構與維運層的負擔搬進來，也沒有把文件寫成一個巨大的神祕黑盒。對 GCD 來說，可讀性、可以在本地測試與容易理解 的優先順序，比「看起來像原版」更高。

舉例來說，如果今天團隊想把「先閱讀 AGENTS.md，再執行 /review，最後必要時跑測試」固定成工作流，Claude Code 系語言會很自然地把這件事理解成 instructions + commands + tools 的組合。GCD 借的正是這種層次感，而不是某個外觀或品牌口吻。

Gemini CLI 對 GCD 的影響，主要在產品邊界感

Gemini CLI 值得學的地方，不只是 terminal-first 這個表面風格，而是它把 context file、trusted folders、MCP、headless output、token caching 這些事情都視為產品核心設計。這提醒了 GCD：AI 助手 不只是在 terminal 裡接一個模型，而是要把工作區關係、輸出型態、擴充介面與成本控制一起想進去。

GCD 目前已吸收 terminal-first、workspace config、provider profiles、MCP awareness 這條思路，但 token caching 與更成熟的 擴充生命週期 仍屬 。這個差距很重要，因為它提醒我們參考不能只借名詞。如果沒有 快取金鑰、請求簽名、失效條件與 provider-specific 路徑，就不能把「有受到啟發」寫成「已有完整 token caching」。

換個更具體的例子來說，Gemini CLI 脈絡會提醒我們：同一個 /review 任務，不只需要 prompt，還可能需要 context file、trusted folder 決策、headless output 與未來的 MCP 工具擴充。這讓 GCD 在設計時不會只把注意力放在模型回覆文字，而會把「執行環境與外部工具邊界」一起納入。

OpenAI Codex 對 GCD 的影響，主要在人機協作節奏

Codex 系列最值得學的，是一種很清楚的本地 AI 助手 UX：instructions 驅動、approval 明確、sandbox 分級、terminal 裡就能完成大部分編碼工作。GCD 的 suggest / auto-edit / full-auto、sandbox policy，以及 plan-only / live mode 的切分，都能看出這條影響線。

但 GCD 並不打算把自己做成單一供應商工具。它把 provider 清單、fallback 與 adapter 放在主要流程，這是一個很有意識的取捨：對社群與學習目的來說，供應商可替換性本身就是產品價值，不是最後才補的相容模式。

一個很好理解的例子是：當模型想直接套 patch 時，Codex 式思路會先問「這次修改能不能被批准、要不要隔離、是否該由使用者確認」。GCD 把這套語言保留了下來，所以它不是把工具呼叫當純技術細節，而是當成與使用者建立信任節奏的一部分。

為什麼 GCD 不應該把三家的 功能清單 全部補齊

參考不是功能集郵

有些能力看起來很吸引人，但如果沒有實際需求、沒有足夠原始碼支撐、沒有清楚的學習價值，就不該因為「別人有」而直接複製。GCD 的定位不是用最少時間變成三家產品的總和，而是做一個可以觀察、可對照、可實際做出來的 外殼 樣本。

GCD 想走自己的路的地方

1

Rust-native orchestration

不是為了語言情懷，而是希望把 提示詞組裝、provider adapters、session storage 做成更容易測與讀的 domain code。

2

多供應商不是附加功能

對社群來說，相容性與成本是 adoption 的現實問題。GCD 把 provider 清單 與 fallback 放得很前面，就是這個判斷的體現。

3

文件先於野心

先把主要流程講清楚，再去談 dream memory、worktree isolation、token caching。這一點更接近教學用的範例系統，而不是追逐話題功能。
