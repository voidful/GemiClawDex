Roadmap

實況檢查 後，GCD 下一步應該補哪裡

先補會影響主要流程的缺口，再談更大的產品野心。這一章故意把「應該先做什麼」講得比「未來可以做什麼」更清楚。

優先順序不是由酷炫程度決定

Roadmap 最容易寫成願望清單，但對教學型專案來說，真正重要的是先補那些會讓讀者與使用者誤判系統能力的缺口。GCD 的下一步，不應該先問「哪個功能最帥」，而是先問「哪個缺口最影響主要流程的可信度、可維護性與可教性」。

把文件與主要流程再對齊

持續校正文實落差，例如 container sandbox 的邊界、memory integration 的現況、MCP 相容度的描述。這能立即降低誤解成本。

如果只能做一個兩週衝刺，應該先做什麼

Roadmap 真正難的地方不是列想法，而是有限工期下怎麼排。如果 GCD 只有一個短衝刺可以投入，我會建議先把會直接改變使用者認知與安全意義的項目排進去。

優先項目 為什麼先做 完成後會改善什麼
Container sandbox 含義補齊或明確收斂 這是最容易被誤解成「已完整隔離」的地方。 安全設計會更誠實，文件也能更明確說清楚哪些工具在哪裡執行。
MCP client 穩定度 它牽涉未來擴充脈絡，但現在含義還偏簡化。 可擴充性會從「可演示」往「可依賴」前進一步。
Memory 與 prompt / session 的正式連接 現在最像有能力名稱、但尚未真正參與主要流程決策。 能把記憶從孤立檔案提升為有工程意義的上下文層。

為什麼這樣排優先順序

原則 對 GCD 的意思 如果忽略它，會出現什麼問題
先補主要流程缺口 container 一致性、MCP 穩定度、memory integration 都比高階功能更值得先做。 文件會愈寫愈難誠實，使用者也會誤判安全與完成度。
先解決解釋成本高的地方 像 規劃模式、工作接續、sandbox 邊界都需要更清楚含義。 讀者會以為功能很多，實際上卻不知道哪些真的能依賴。
把整合成本當成核心設計 token caching、請求格式調整、provider fallback 不是附帶優化，而是可擴充性的基礎。 每接一個 provider 或新能力，都會重新付一次不必要的工程成本。
最後才做高階玩法 Dream-style memory consolidation、worktree isolation、更多 sub-agent 協調應建立在穩主要流程之後。 系統表面上變強，底層卻越來越難測、難教、難維護。

什麼事情故意不該先做

N1

先做超多高階模式

在 container、MCP、memory 這些邊界仍模糊時，太早衝 multi-agent / dream pass，只會讓主要流程更難解釋。

N2

先補齊三大產品所有 feature 名稱

這會把 roadmap 變成功能集郵，而不是工程問題排序。

N3

先把文件寫得比實作更滿

對 教學範例 來說，這是最危險的債，因為一旦文實脫節，整站可信度會快速下滑。

具體的工程重點

主題 為什麼重要 下一步建議
Container sandbox 一致性 現在只有 shell 真正進容器，容易讓使用者誤以為所有操作影響都被隔離。 明確定義哪些工具能進容器，哪些必須留在 host，並同步更新 UI 與文件。
MCP client 穩定度 MCP 是 GCD 向外擴充的重要接口，但目前描述應維持保守。 補強 framing、錯誤處理、server capability negotiation，再擴大文件承諾。
Memory integration MEMORY.md 如果不回傳到 prompt / sessions，就只是孤立檔案。 先定 memory read/write 進入哪條主要流程，再談 consolidation 或 dream pass。
Token caching 與 請求格式調整 這會直接影響成本、速度與 provider 切換穩定性。 先定 快取金鑰 與 請求簽名 策略，再做 provider-specific 實作。
Multimodal attachment path 目前附件 metadata、prompt placeholder 與 exec JSON 已存在，但 provider adapter 對二進位附件仍主要停在文字主要流程。 從單一 provider 的圖片 / PDF 路徑開始做完整連接，並同步補 replay / session 可追蹤證據。
Hook / sub-agent 政策證據模型 現在已有 hook deny 與 有限制的委派，但事件層仍不易看出拒絕來源，sub-agent 也還不是第一級 session 工作單元。 把拒絕來源、hook 層級與 sub-agent lineage 補進事件與 replay 證據，再決定是否升級為更完整的工作代理模型。
Worktree isolation 這會改變 git、session、sandbox 與路徑策略，是大工程。 先把現有工作區內的 trust / sandbox 邏輯穩住，再決定是否導入。

GCD 目前較接近版本 A 到 B 之間的過渡態

如果把 roadmap 換成教學語言，它其實在說什麼

第一，先把「會被誤會成完整能力」的部分說清楚

像 container sandbox、memory、MCP 都不是不能做，而是現在更需要先把邊界講清楚，避免設計語言先跑到實作前面。

第二，讓多 provider 與整合成本進入核心設計

這會決定 GCD 未來是「每接新模型就重做一遍」還是「真正有一個穩定的 外殼 底座」。

第三，只有在主要流程清楚後，高階能力才有學習價值

如果底層邊界還沒穩，太早談 dream memory 或 worktree isolation，讀者只會得到一個看似雄心勃勃但難以實作的願景。

如果把 roadmap 當成教學地圖，下一輪章節最值得補哪裡

延伸 container 章到真正的 worktree / snapshot 設計

現在已經把 shell 容器化邊界講清楚；下一步如果要再升級，就要處理工作區複製、git 回寫與 session 含義。

把 provider 解剖室往 快取金鑰 / retry 策略再推深一層

請求格式調整 與 fallback 案例已補上；下一步最值得寫的是 快取金鑰、header 差異與更細的 retry 策略，讓整合成本不只停在協定層。

把 Runtime Guardrails 章再往 sub-agent / 成本策略推深

現在已經把 context compaction、fallback、token accounting、undercover auto-git 講清楚；下一步最值得補的是 cost-aware fallback、provider family 策略與更精細的 budget 策略。

把政策插點章再往拒絕來源 / 子任務血緣推深

現在已經把 hooks、permission prompt、sub-agent delegation 的現況講清楚；下一步最值得補的是事件層拒絕來源、政策設定檔與 replay 中的子任務血緣。

把工作流語言章再往命令衝突 / skill 指派 推深

現在已經把 instructions、commands、skills 的分工講清楚；下一步最值得補的是重名衝突規則、catalog 優先順序與未來是否導入自動 skill 指派。

把 Attachment 章再往 provider-native upload / replay persistence 推深

現在已經把 binary reference、prompt placeholder、exec JSON 與 adapter 斷點講清楚；下一步最值得補的是單一 provider 的真正 payload 與 session 證據模型。

把 Trust Matrix 再往 MCP / memory / session 的統一含義推深

現在已把 instructions、commands、skills、providers、plugins、MCP、session、memory 的 trust 差異講清楚；下一步最值得補的是把例外路徑逐步收斂成更一致的 gate 規則。
