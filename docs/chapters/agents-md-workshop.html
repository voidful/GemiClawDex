AGENTS.md 撰寫工作坊

Harness Engineering 最小成本、最大效果的起點。學完這一章，你就能為自己的專案寫出一份有效的 AGENTS.md。

為什麼 AGENTS.md 是第一步

Anthropic 的實測顯示：同一個模型（Opus 4.5），沒有 harness 時花 $9 和 20 分鐘產出無法使用的結果。有完整 harness 時花 $200 和 6 小時產出可運行的程式。而 AGENTS.md 是 harness 中成本最低、效果最明顯的部分，因為它只是一個文字檔。

AGENTS.md 在 Harness 分類中的位置

Martin Fowler 的 Harness 分類把所有 harness 組件分成四個象限。AGENTS.md 屬於 Inferential Guide：它是一份用自然語言寫的指引，告訴 agent 應該怎麼在這個專案裡工作。

Guide（引導 agent 的行為）Sensor（檢查 agent 的輸出）
Computational（機器可執行） Linter 規則、Type checker、CI config Test suite、Structural tests
Inferential（需要 LLM 理解） ✅ AGENTS.md、參考文件、範例程式碼 LLM-as-judge、AI code review

拆解 GCD 自己的 AGENTS.md

GCD 的 AGENTS.md 是一份活教材。以下逐段解釋為什麼這樣寫：

AGENTS.md（節錄） 
# GemiClawDex — Development Instructions

## Project Purpose
GCD is a teaching project for Harness Engineering.

## Coding Conventions
1. Keep gcd-core free of unnecessary dependencies.
2. Prefer plain-text, testable domain logic.
3. Make trust and sandbox decisions explicit in the output.
4. Treat command loading, skill loading, and prompt assembly as separate layers.
5. Preserve offline testability.

## What NOT to Do
- Do not add features just because they are "cool."
- Do not write documentation that claims a feature is "fully implemented" when it is a stub.
段落為什麼這樣寫效果
Project Purpose 讓 agent 知道這是教學專案，不是生產工具 agent 會優先考慮可讀性，而不是極致效能
Coding Conventions 具體的約束比抽象原則更有效 agent 不會引入不必要的依賴
What NOT to Do 禁止項比建議項更有效。agent 很擅長遵守禁令 避免功能膨脹和狀態誤標
File Dependency Chain 讓 agent 知道改一個檔案會影響哪些其他檔案 減少遺漏的連帶修改

寫好 AGENTS.md 的五個原則

1

告訴 agent「你在哪」

專案結構、技術棧、關鍵檔案路徑。Agent 不會自己探索整個 repo，你要告訴它重點在哪。

2

告訴 agent「不要做什麼」

禁止項比建議項更有效。「不要引入新依賴」比「盡量減少依賴」清楚十倍。

3

告訴 agent「怎麼驗證」

測試指令、lint 指令、格式檢查指令。讓 agent 可以自己確認工作品質。

4

保持短小

AGENTS.md 的內容會佔用 context window。500 字以內是理想長度。太長的話，用子目錄的 AGENTS.md 分層放置。

5

分層放置

根目錄放全域規範。子目錄放局部規則。GCD 的 instructions.rs 會自動合併多層 AGENTS.md。

三個範例

範例 A：Python Flask 專案

AGENTS.md 
# My Flask App

Python 3.12 + Flask 3.x + SQLAlchemy 2.x.

## Structure
- app/ — Flask application
- tests/ — pytest tests
- migrations/ — Alembic migrations

## Rules
- All endpoints must have type hints.
- All database queries must use SQLAlchemy ORM, not raw SQL.
- Run `pytest` before committing. All tests must pass.
- Do not modify migration files directly. Use `flask db migrate`.

## Testing
```bash
pytest
pytest --cov=app
flask db upgrade  # before running tests
```

範例 B：Monorepo

AGENTS.md（根目錄） 
# Acme Platform

Monorepo: frontend (Next.js) + backend (Go) + shared (protobuf).

## Cross-cutting rules
- Frontend changes must not break backend API contracts.
- Shared proto changes require regenerating both frontend and backend stubs.
- Each package has its own AGENTS.md with local rules.

## CI
```bash
make lint    # runs all linters
make test    # runs all tests
```

範例 C：資料科學專案

AGENTS.md 
# Sentiment Analysis Pipeline

Python 3.11 + pandas + scikit-learn + DVC for data versioning.

## Rules
- Never commit data files. Use DVC.
- Notebooks are for exploration only. Production code goes in src/.
- All model training must be reproducible: fix random seeds, log hyperparameters.

## Testing
```bash
pytest tests/
python -m src.train --dry-run  # verify pipeline without GPU
```
